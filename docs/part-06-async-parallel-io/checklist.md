# Part 6 レビュー用チェックリスト

非同期、並列、外部 I/O を含む変更をレビューするときは、速さだけでなく、待機、中断、失敗、同時実行数、外部境界を確認する。特に「正常時は動く」コードほど、キャンセル、タイムアウト、部分失敗、リトライ時の副作用を明示する。

## async / await

- [ ] I/O-bound の待機を同期ブロックせず、呼び出しチェーン全体が `async` / `await` でつながっている。
- [ ] `.Result`、`.Wait()`、`GetAwaiter().GetResult()` を使っていない。使う場合は同期境界として理由を説明できる。
- [ ] `async void` はイベントハンドラーなど、呼び出し元が `Task` を待てない場所に限定されている。
- [ ] 非同期メソッド名、戻り値、引数から、待機とキャンセルの扱いが分かる。
- [ ] 例外を握りつぶさず、呼び出し元が失敗を判断できる形で返している。
- [ ] `Task.Run` で同期 I/O を包んで、非同期 API がない問題を隠していない。
- [ ] fire-and-forget が必要な場合、例外ログ、キャンセル、アプリ終了時の扱いが明示されている。
- [ ] `Task.WhenAll` の例外集約と部分成功時の扱いを、呼び出し元が判断できる。

## Task、ValueTask、CancellationToken

- [ ] 公開 API や時間がかかる処理に、必要な `CancellationToken` が用意されている。
- [ ] 受け取った `CancellationToken` を、HTTP、DB、delay、channel など下位 API に渡している。
- [ ] キャンセル、タイムアウト、業務エラーを同じ例外変換やリトライ対象にしていない。
- [ ] キャンセル時に、永続化、送信、状態更新が中途半端に残らない境界を決めている。
- [ ] `ValueTask` は、実測または低レイヤー API としての明確な理由がある場合だけ使っている。
- [ ] 完了済み `Task`、`Task.FromResult`、`Task.CompletedTask` の使い分けが意図に合っている。
- [ ] `CancellationToken.None` や `default` を渡して、上位からの中断要求を意図せず切っていない。
- [ ] timeout 用 token と呼び出し元の token を `CreateLinkedTokenSource` などで必要に応じて結合している。
- [ ] キャンセル可能な処理では、キャンセル後にドメインイベント送信や外部 API 呼び出しなどの副作用を続けない。
- [ ] `OperationCanceledException` を一般的な失敗ログやアラートとして扱い、運用ノイズにしていない。

## ConfigureAwait

- [ ] 対象コードがアプリケーションコードか、再利用されるライブラリコードかを分けて判断している。
- [ ] `await` 後に UI スレッドやリクエストコンテキストへ戻る必要があるか確認している。
- [ ] プロジェクト内の `ConfigureAwait(false)` 方針と一貫している。
- [ ] 同期ブロックを残したまま、`ConfigureAwait(false)` だけでデッドロック対策にしていない。
- [ ] `ConfigureAwait(false)` を性能改善として過大評価していない。

## 並列処理と同時実行数

- [ ] 対象処理が CPU-bound か I/O-bound かを分けて、`Parallel`、PLINQ、`Task.WhenAll`、`Channel` を選んでいる。
- [ ] 同時実行数に上限があり、外部 API、DB、スレッドプール、メモリを過剰に使わない。
- [ ] 共有状態へのアクセスが、`lock`、不変データ、thread-safe collection、メッセージパッシングなどで保護されている。
- [ ] 並列化による順序の変化が、業務結果、ログ、テストに影響しないか確認している。
- [ ] 例外、キャンセル、部分成功時に、投入済み作業をどう扱うか決まっている。
- [ ] CPU-bound ではない処理を `Task.Run` で包んで、問題を見えにくくしていない。
- [ ] `Task.WhenAll(items.Select(...))` が入力件数ぶん一気に外部 I/O を開始しないよう、上限を置いている。
- [ ] `Parallel.ForEachAsync`、`SemaphoreSlim`、`Channel` などの上限値が、DB 接続数や外部 API 制限と矛盾していない。
- [ ] producer / consumer では backpressure、完了通知、例外時の channel close が設計されている。
- [ ] 並列処理内で `List<T>`、`Dictionary<TKey,TValue>`、乱数、日時、集計値などの共有 mutable state を無防備に触っていない。

## HttpClient と外部 I/O

- [ ] `HttpClient` の生成とライフタイムが、`IHttpClientFactory` などで管理されている。
- [ ] BaseAddress、timeout、headers、認証、retry policy がクライアント単位で整理されている。
- [ ] HTTP ステータスごとの扱いが業務要件と一致し、成功扱い、再試行、ユーザー向けエラーを分けている。
- [ ] JSON の null、形式不正、未知プロパティ、互換性のない変更を考慮している。
- [ ] 外部 API 用 DTO とドメインモデルの境界があり、外部仕様を業務コードへ直接漏らしていない。
- [ ] ログに URL、ステータス、相関 ID など調査に必要な情報があり、機密情報は出していない。
- [ ] `new HttpClient()` を呼び出しごとに作らず、socket 枯渇や DNS 更新の扱いをライフタイム設計に含めている。
- [ ] `HttpResponseMessage` や stream を適切に dispose し、大きなレスポンスを不用意に全量読み込みしていない。
- [ ] `EnsureSuccessStatusCode()` だけで業務エラー本文や 404 / 409 / 429 などの意味を失っていない。
- [ ] request ごとの timeout、全体 timeout、リトライを含む deadline が混ざっていない。
- [ ] 認証ヘッダーや correlation ID を static な default header に入れて、リクエスト間で漏らしていない。
- [ ] typed client / named client の責務が外部 API 単位にまとまり、業務 service が HTTP の詳細を直接持っていない。

## タイムアウト、リトライ、キャンセル

- [ ] キャンセルは呼び出し元の意思、タイムアウトは待機時間の境界として分けて扱っている。
- [ ] リトライ対象の例外、HTTP ステータス、回数、待機時間、最大待機時間が明確になっている。
- [ ] リトライ待機にも `CancellationToken` が伝播している。
- [ ] 副作用のある処理では、冪等性キー、重複検知、再送時の整合性を確認している。
- [ ] timeout、retry、circuit breaker などの方針がクライアントやユースケースごとに一貫している。
- [ ] 過剰なリトライで障害先や自システムへ負荷を増やさない設計になっている。
- [ ] 4xx、認証エラー、入力不正、キャンセルなど、再試行しても成功しない失敗を retry 対象から外している。
- [ ] 429 / 503 などでは `Retry-After`、指数バックオフ、jitter、最大待機時間を検討している。
- [ ] リトライ全体の所要時間が、呼び出し元の SLA や HTTP request timeout を超えない。
- [ ] リトライごとに同じ副作用を重ねる処理では、冪等性キーや外部側の重複排除を確認している。
- [ ] retry policy のログが試行ごとに過剰なエラーを出さず、最終失敗と途中経過を区別できる。

## テストと観測性

- [ ] 正常系だけでなく、キャンセル、タイムアウト、リトライ対象外エラー、部分失敗のテストがある。
- [ ] 外部 I/O は専用クライアントや adapter に閉じ込められ、テストで差し替えられる。
- [ ] 並列処理のテストが、偶然の実行順序や固定の sleep に依存していない。
- [ ] 失敗時のログ、メトリクス、例外情報から、どの外部呼び出しがどの理由で失敗したか追える。
- [ ] タイムアウト値や同時実行数が、設定または運用上説明できる場所に置かれている。
- [ ] retry、timeout、circuit breaker のテストが、実時間の長い sleep に依存せず、fake clock や短い delay で検証できる。
- [ ] 並列度や retry 回数を設定変更したとき、負荷や外部 API 制限に対する影響をレビューできる。

## レビューコメントの観点

- [ ] 「非同期にすべき」ではなく、スレッド占有、キャンセル、失敗伝播、同時実行数のどれを改善する指摘かを明確にしている。
- [ ] 「速くなるはず」ではなく、CPU-bound / I/O-bound、測定値、外部制約のどれを根拠にしている。
- [ ] リトライや並列化を提案する場合、副作用、順序、上限、観測性も合わせて確認している。
- [ ] `CancellationToken` の指摘では、API 表面、下位 API への伝播、キャンセル後の副作用、ログ分類を分けて書いている。
- [ ] `HttpClient` の指摘では、生成場所、ライフタイム、timeout、status handling、DTO 境界のどれが問題かを明確にしている。
