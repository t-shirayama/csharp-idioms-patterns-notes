# 例外を制御フローに使わない

## 概要

例外は想定外の失敗を呼び出し元へ伝えるための仕組みです。通常起こりうる入力不正や検索失敗を例外で表すと、コストが高くなるだけでなく、正常系と異常系の読み分けが難しくなります。

```csharp
// 避けたい例: 通常の入力不正を例外で分岐している。
try
{
    var age = int.Parse(input);
    Register(age);
}
catch (FormatException)
{
    ShowValidationError();
}
```

```csharp
if (int.TryParse(input, out var age))
{
    Register(age);
}
else
{
    ShowValidationError();
}
```

## 判断基準

- ユーザー入力の不正、検索結果なし、在庫なしなど通常起こる失敗は戻り値で表す。
- DB 障害、ネットワーク断、整合性破壊など想定外の失敗は例外として扱う。
- API 境界では、domain error、validation error、unexpected error を分けて応答へ変換する。
- 例外を捕まえる場所は、ログ、トランザクション、ユーザー応答を決められる境界に寄せる。
- 例外を握りつぶす場合は、呼び出し元から見た成功条件が変わっていないか確認する。

## 使いどころ

- `TryParse` pattern は、失敗が通常起こりうる入力処理に向いている。
- validation error は、画面や API の応答として返せる形にする。
- unexpected error は例外として扱い、ログや監視に載せる。

## 避けたい書き方

- 検索結果なし、入力形式不正、権限なしなどの expected error をすべて例外にする。
- `catch (Exception)` で握りつぶし、呼び出し元に成功したように見せる。
- 例外の発生を前提にしたループやリトライを書く。
- 低レイヤーで例外を UI 文言へ変換し、再利用性を落とす。

```csharp
// 避けたい例: 見つからないことが通常ケースなら例外でなく戻り値にする。
public User GetByEmail(string email)
{
    return users.Single(user => user.Email == email);
}
```

```csharp
public User? FindByEmail(string email)
{
    return users.SingleOrDefault(user => user.Email == email);
}
```

```csharp
public Result<Order> Place(OrderRequest request)
{
    if (request.Items.Count == 0)
    {
        return Result.Invalid("明細が必要です。");
    }

    return Result.Success(orderFactory.Create(request));
}
```

## レビュー観点

- その失敗は expected error か、unexpected error か。
- 呼び出し元が分岐すべき失敗を戻り値、Result 型、validation result で表現できないか。
- 例外を捕まえる場所が境界に寄っているか。低レイヤーで握りつぶしていないか。
- ログに出すべき例外と、ユーザーへ返す validation error を混同していないか。
- `catch (Exception)` の範囲が広すぎず、再 throw やログの方針が明確か。
- 例外メッセージに個人情報やシークレットが含まれていないか。
