# Part 8: テストしやすい C#

テストしやすい C# は、テストコードのためだけに作るものではありません。副作用の境界、依存の向き、入力と出力の明確さを整えることで、仕様変更や障害調査のときにも判断しやすいコードになります。

実務では、すべてを mock できる形にするよりも、「業務判断を小さく純粋に保ち、時刻、乱数、外部 I/O など変動するものを境界で扱う」ことを優先します。テストは設計の結果であり、設計の違和感を早く見つける道具でもあります。

## このPartで身につくこと

- 単体テストしやすい責務分割と依存の向き
- xUnit / NUnit / MSTest を選ぶときに見る観点
- mock、stub、fake を目的で使い分ける判断
- 時刻、乱数、外部 I/O を安定してテストするための分離
- テストデータを読みやすく保つ Builder パターン
- 例外、非同期、キャンセル、境界値をテストに落とす考え方
- テストが「何を保証しているか」をレビューで説明する力

## 重要トピック

- テストしやすさは public メソッドの数ではなく、副作用と依存の扱いで決まる
- 業務判断はできるだけ純粋な入力と出力に寄せる
- mock は振る舞いの確認、stub は入力の固定、fake は軽量な代替実装として使い分ける
- 時刻、乱数、外部 API、ファイル、DB は境界を作り、テストで制御できるようにする
- テストデータは意図が見える最小差分で組み立てる
- 非同期テストでは await、キャンセル、タイムアウト、例外伝播を明示する
- 境界値テストでは「ちょうど境界」と「境界を一つ超えた値」を確認する

## 実務での使いどころ

仕様変更が多い業務ロジックでは、条件分岐を外部 I/O から切り離してテストできるようにしておくと、変更の影響を小さく確認できます。DB や API を直接叩くテストだけに頼ると、失敗原因が仕様なのか環境なのか判別しにくくなります。

障害修正では、再現条件をテストとして残すことで同じ問題の再発を防げます。特に時刻、タイムゾーン、乱数、リトライ、非同期キャンセル、境界値は、実装者の手元では再現しにくいため、テスト側で条件を固定できる設計が重要です。

## よくある落とし穴

- テストのために private メソッドを直接テストしようとする
- すべてを mock にし、実際の組み合わせで起きる問題を見落とす
- mock の呼び出し回数だけを確認し、結果や業務上の意味を確認していない
- `DateTime.Now`、乱数、ファイル、HTTP、DB がコード中に直書きされている
- テストデータの準備が長く、何を確認したいテストか読めない
- 非同期メソッドを await せず、失敗がテストに伝わらない
- 正常系だけを確認し、例外、空、上限、下限、キャンセルを見ていない

## レビュー観点

- テスト対象の業務判断が外部 I/O から切り離されているか
- mock、stub、fake の選択が、確認したいことに合っているか
- 時刻、乱数、外部 API、ファイル、DB をテスト側で制御できるか
- テスト名、Arrange、Act、Assert から仕様が読めるか
- テストデータが必要な差分だけを表し、余計な値で意図を隠していないか
- 例外、非同期、キャンセル、境界値のテストが抜けていないか
- テストが壊れたとき、仕様変更、実装バグ、環境問題のどれかを切り分けやすいか
