# Part 8: テストしやすい C#

テストしやすい C# は、テストコードのためだけに作るものではありません。副作用の境界、依存の向き、入力と出力の明確さを整えることで、仕様変更や障害調査のときにも判断しやすいコードになります。

実務では、すべてを mock できる形にするよりも、「業務判断を小さく純粋に保ち、時刻、乱数、外部 I/O など変動するものを境界で扱う」ことを優先します。テストは設計の結果であり、設計の違和感を早く見つける道具でもあります。
