# 単体テストしやすい設計

## 概要

単体テストしやすい設計では、計算や判断を副作用から切り離します。入力が同じなら結果も同じになる pure function は、テストが短く、失敗理由も追いやすくなります。一方で DB、HTTP、ファイル、現在時刻などに直接依存する処理は、テストの準備が重くなりがちです。

## 使いどころ

- 金額計算、状態遷移、入力検証など、業務判断を小さなメソッドや型に閉じ込める。
- 外部依存は constructor injection で渡し、呼び出し側から差し替えられるようにする。
- static メソッドは、状態を持たない純粋な処理か、薄い入口に限定する。

```csharp
public sealed class DiscountCalculator
{
    public decimal Calculate(decimal subtotal, bool isPreferredCustomer)
    {
        var rate = isPreferredCustomer ? 0.1m : 0m;
        return subtotal * (1 - rate);
    }
}
```

## 設計の判断基準

テストしやすさは、mock できることよりも「何を入力し、何を期待するか」が明確であることです。

- 業務判断は入力値と戻り値で確認できる形にする。
- DB、HTTP、ファイル、時刻、乱数は境界に寄せる。
- 1 つの public method が複数の理由で失敗するなら、責務の分割を検討する。
- private method を直接テストしたくなる場合は、隠れた概念がないか見る。
- DI は差し替える価値のある依存に使い、値や純粋計算まで interface にしない。

```csharp
public sealed class OrderPolicy
{
    public bool CanSubmit(Order order)
    {
        return order.Items.Count > 0 && order.Status is OrderStatus.Draft;
    }
}
```

このような判断は、repository や logger を使わずにテストできます。usecase はこの policy を呼び、保存や通知の手順を担当します。

## 副作用を端に寄せる

副作用が混ざる処理では、業務判断と実行を分けます。例えば「送信すべきメールを決める」と「実際にメールを送る」を分けると、前者は単体テスト、後者は adapter の統合テストにできます。

```csharp
public sealed record MailCommand(string To, string Subject);

public MailCommand CreateWelcomeMail(User user)
{
    return new MailCommand(user.Email, "Welcome");
}
```

戻り値として「これから行う副作用」を表すと、テストでは値を確認できます。実行部分は薄く保ち、失敗時のリトライやログは境界側で扱います。

## 避けたい書き方

- 業務判断の中で `DateTime.Now`、`Guid.NewGuid()`、DB 更新などを直接呼ぶ。
- private メソッドを無理にテストしたくなるほど、1 つの public メソッドに責務を詰め込む。
- テストしやすさだけを理由に、意味のない interface を大量に作る。
- テストから内部状態を覗くためだけの public property を増やす。
- singleton や static mutable state により、テストの実行順で結果が変わる。

```csharp
// 避けたい例: 現在時刻に直接依存しており、期限切れの境界を固定してテストしにくい。
public bool IsExpired(DateTime expiresAt)
{
    return expiresAt <= DateTime.Now;
}
```

```csharp
// 避けたい例: 外部 I/O と判断が密結合している。
public async Task<bool> CanSubmitAsync(Guid orderId)
{
    var order = await dbContext.Orders.FindAsync(orderId);
    return order is { Status: OrderStatus.Draft } && order.Items.Any();
}
```

## テスト対象の選び方

すべてのクラスを同じ粒度で単体テストする必要はありません。純粋な業務判断は細かく、DB や HTTP の adapter は本物に近い統合テストで確認する、という分け方が現実的です。

- domain policy: 境界値や状態遷移を細かくテストする。
- usecase: 成功、業務エラー、外部依存失敗の手順をテストする。
- repository: SQL、mapping、transaction は DB を使ったテストを検討する。
- controller: routing、status code、validation response を薄く確認する。

## レビュー観点

- 業務ルールが外部 I/O と混ざらず、入力と出力で検証できるか。
- constructor injection で渡す依存が、本当に差し替える価値のある境界か。
- static 依存や singleton が、テスト順序や並列実行に影響しないか。
- テストのための公開 API が増えすぎていないか。
- テストで確認したい仕様が、mock の呼び出し順ではなく観測可能な結果になっているか。
- 単体テストと統合テストの役割分担が、外部依存のリスクと合っているか。
