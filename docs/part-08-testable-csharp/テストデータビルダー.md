# テストデータビルダー

## 概要

テストデータビルダーは、テストに必要なオブジェクトを読みやすく組み立てるためのパターンです。object mother のような固定データは手軽ですが、ケースごとの差分が増えると意図が見えにくくなります。builder は「既定値は妥当、テストで重要な値だけ明示する」形にすると変更に強くなります。

## 使いどころ

- 必須プロパティが多い entity や command を、毎回長く初期化したくない。
- テストごとに一部の値だけ変えて、条件を読みやすくしたい。
- ドメインモデルの constructor 変更に合わせて、テスト全体を大量修正したくない。

```csharp
public sealed class OrderBuilder
{
    private Guid customerId = Guid.Parse("aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa");
    private decimal total = 1000m;

    public OrderBuilder ForCustomer(Guid value)
    {
        customerId = value;
        return this;
    }

    public OrderBuilder WithTotal(decimal value)
    {
        total = value;
        return this;
    }

    public Order Build() => new(customerId, total);
}
```

## 既定値の決め方

builder の既定値は「多くのテストで意味を持つ正常値」にします。不正値を既定にすると、各テストが毎回補正を強いられ、arrange が読みにくくなります。

```csharp
public sealed class UserBuilder
{
    private string displayName = "Test User";
    private string email = "user@example.com";

    public UserBuilder WithEmail(string value)
    {
        email = value;
        return this;
    }

    public User Build() => User.Register(displayName, email);
}
```

builder は本番コードの constructor や factory を通す方が安全です。private setter や reflection で不変条件を迂回すると、テストだけで存在する不正な状態を作り、本番の安心につながりません。

## 意図が見える arrange

テストでは、重要な値だけを明示します。すべてのプロパティを毎回並べるより、builder のメソッド名で仕様を表す方が読みやすいことがあります。

```csharp
var order = new OrderBuilder()
    .WithTotal(999m)
    .ForPreferredCustomer()
    .Build();
```

境界値や異常値は、テスト名と builder の呼び出しの両方から意図が分かるようにします。`WithTotal(-1)` だけでは、なぜ負数を渡しているのかが後から分かりにくい場合があります。

```csharp
var order = new OrderBuilder()
    .WithInvalidNegativeTotal()
    .BuildWithoutValidation();
```

このような不正状態を作るメソッドは、検証対象をはっきりさせる場合に限ります。通常のテストで多用すると、domain の不変条件を壊すテストが増えます。

## object mother との使い分け

object mother は、代表的なシナリオ名を持つ固定データに向いています。差分を細かく変えるテストでは builder の方が読みやすいです。

```csharp
public static class Orders
{
    public static Order SubmittedOrder()
    {
        return new OrderBuilder()
            .Submitted()
            .Build();
    }
}
```

object mother が巨大化して、どのテストがどの値に依存しているか分からなくなったら、builder へ寄せる合図です。

## 避けたい書き方

- すべてのテストが巨大な共通 object mother に依存し、どの値が重要か分からない。
- builder の既定値が不正な値で、各テストが毎回補正を強いられる。
- テスト名や arrange が曖昧で、境界値や異常値を使う理由が読み取れない。
- builder が本番の不変条件を常に迂回し、存在しない状態を量産する。
- fluent API が増えすぎて、builder 自体の仕様理解が必要になる。

```csharp
// 避けたい例: 重要でない値まで毎回露出している。
var user = new User(
    Guid.NewGuid(),
    "Test User",
    "user@example.com",
    false,
    DateTimeOffset.UtcNow);
```

## レビュー観点

- テストで重要な値だけが明示され、残りは妥当な既定値に任せているか。
- builder が本番コードの不変条件を迂回していないか。
- object mother、fixture builder、inline 初期化の使い分けが複雑になっていないか。
- テスト名がデータの意図を補っているか。
- 不正データを作る builder メソッドが、検証目的を明確にした名前になっているか。
- builder の変更が多くのテストの意味を暗黙に変えてしまわないか。
