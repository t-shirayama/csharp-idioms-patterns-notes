# タイムアウト、リトライ、キャンセル設計

## 概要

外部サービスは遅延し、失敗し、ときには部分的に成功します。タイムアウトは待ち続けないための境界、リトライは一時的な失敗から回復するための手段、キャンセルは呼び出し元の中断意思です。この 3 つを混同すると、二重実行、長すぎる待機、原因不明の失敗が起きやすくなります。

```csharp
public async Task<InvoiceStatus> GetStatusWithRetryAsync(
    string invoiceId,
    CancellationToken cancellationToken)
{
    for (var attempt = 1; attempt <= 3; attempt++)
    {
        try
        {
            return await invoiceClient.GetStatusAsync(invoiceId, cancellationToken);
        }
        catch (HttpRequestException) when (attempt < 3)
        {
            await Task.Delay(TimeSpan.FromMilliseconds(200 * attempt), cancellationToken);
        }
    }

    throw new InvalidOperationException("Retry loop exited unexpectedly.");
}
```

## 使いどころ

- ネットワーク断、503、429 など、一時的な失敗が想定される外部呼び出し。
- 参照系や冪等な更新など、再実行しても意味が壊れない処理。
- 呼び出し元、アプリケーション、外部クライアントのどこがタイムアウト責務を持つかを決める場面。
- バッチやバックグラウンド処理で、全体の締め切りと 1 回ごとの待機上限を分けたい。

```csharp
using var timeout = CancellationTokenSource.CreateLinkedTokenSource(cancellationToken);
timeout.CancelAfter(TimeSpan.FromSeconds(30));

var result = await invoiceClient.GetStatusAsync(invoiceId, timeout.Token);
```

## 判断基準

タイムアウトは「どれだけ待つか」の境界です。HTTP クライアント、DB コマンド、アプリケーション全体など複数の層で設定できますが、どの層の timeout が発火したか分からない設計にすると調査が難しくなります。

リトライは、一時的に失敗した処理を再実行する設計です。参照系や冪等な更新には向きますが、副作用のある処理では冪等性キー、重複検知、外部サービス側の仕様が必要です。

キャンセルは呼び出し元の意思です。キャンセルされた処理をリトライしてしまうと、ユーザーや上位ジョブが止めたい処理を続けることになります。

## 避けたい書き方

- POST など副作用のある処理を、冪等性キーなしで無条件リトライする。
- `catch (Exception)` であらゆる失敗をリトライする。
- タイムアウトの外側でさらに長いリトライを重ね、ユーザーや上位ジョブの待機時間を超える。
- リトライ待機に jitter を入れず、障害時に全インスタンスが同じタイミングで再試行する。

```csharp
// 避けたい例: キャンセルまでリトライ対象にしてしまう。
catch (Exception)
{
    await Task.Delay(TimeSpan.FromSeconds(1), CancellationToken.None);
}
```

```csharp
// 避けたい例: 副作用のある作成処理を無条件に再実行している。
for (var attempt = 0; attempt < 3; attempt++)
{
    await paymentClient.CreateChargeAsync(request, cancellationToken);
}
```

## よくある失敗

- 1 回ごとの timeout は短いが、リトライ全体の最大時間を決めていない。
- `OperationCanceledException` が timeout 由来か呼び出し元キャンセル由来か区別できない。
- リトライ対象に 400 系の恒久的な入力エラーまで含め、失敗を長引かせる。
- リトライで成功した場合のログやメトリクスがなく、外部サービスの劣化に気づけない。
- 部分成功後の再実行で、二重登録、二重送信、在庫二重引き当てが起きる。

## レビュー観点

- リトライ対象の例外やステータスコードが限定されているか。
- 副作用のある処理に、冪等性キーや重複検知の設計があるか。
- `CancellationToken` がリトライ待機時間にも伝播しているか。
- タイムアウト、リトライ回数、全体の最大待機時間が運用上説明できるか。
- timeout、retry、circuit breaker などの方針がクライアントごとに一貫しているか。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
