# 文字列処理のコスト

## 概要

`string` は immutable です。連結や置換を行うたびに新しい文字列が作られます。少数の連結では問題になりませんが、ループ内で繰り返すと allocation とコピーが増えます。

文字列処理では、コストだけでなく culture も重要です。表示用の整形と、ログ、ID、プロトコル、永続化向けの機械的な整形は分けて考えます。

```csharp
// 避けたい例: ループのたびに新しい string が作られる。
var csv = "";

foreach (var item in items)
{
    csv += item.Code + ",";
}
```

```csharp
var csv = string.Join(",", items.Select(item => item.Code));
```

## 判断基準

- 固定個数の短い連結は、interpolation や `+` で十分なことが多い。
- ループで増やす文字列は、`StringBuilder`、`string.Join()`、`ValueStringBuilder` 相当の選択肢を検討する。
- 比較目的なら `ToLower()` / `ToUpper()` で新しい文字列を作らず、`StringComparison` を指定する。
- 表示用はユーザーの culture、保存やプロトコル用は `InvariantCulture` を明示する。
- ログの文字列は、ログレベルが無効なときにも整形されていないか確認する。

## 使いどころ

- 短い固定個数の連結は interpolation や `+` で十分。
- ループで組み立てる場合は `StringBuilder`、区切り結合なら `string.Join()` を検討する。
- 文化依存の表示には `CultureInfo`、機械的な保存や比較には `CultureInfo.InvariantCulture` を使う。

## 避けたい書き方

- 大量ループ内で `+=` による文字列連結を続ける。
- 表示用フォーマットと永続化用フォーマットで culture を曖昧にする。
- `ToLower()` して比較するなど、余計な文字列を作りつつ culture の問題も招く。
- 例外メッセージやログのために、通常通らない経路でも重い文字列整形を先に実行する。

```csharp
// 比較目的なら、変換してから比べるより comparison を指定する。
if (status.Equals("active", StringComparison.OrdinalIgnoreCase))
{
    Activate();
}
```

```csharp
var text = amount.ToString("N0", CultureInfo.CurrentCulture);
var persisted = amount.ToString(CultureInfo.InvariantCulture);
```

```csharp
if (logger.IsEnabled(LogLevel.Debug))
{
    logger.LogDebug("payload={Payload}", JsonSerializer.Serialize(payload));
}
```

## レビュー観点

- 文字列生成がループ内、ログ大量出力、シリアライズ処理に入っていないか。
- 表示用か、保存用か、比較用かによって culture と comparison が選ばれているか。
- interpolation の中で高コストな処理や副作用を呼んでいないか。
- 可読性のための interpolation と、性能が必要な `StringBuilder` の使い分けが妥当か。
- `string.Join()` の入力が遅延実行で、意図せず複数回列挙されていないか。
- 文字列最適化が必要な箇所か、測定や入力規模で説明できるか。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
