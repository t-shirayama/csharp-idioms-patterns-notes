# Part 14: パフォーマンスと落とし穴

パフォーマンス改善は、速そうな書き方を選ぶ作業ではなく、遅さの原因を測り、影響が大きい場所から直す作業です。実務では、allocation、boxing、文字列処理、LINQ、async、例外処理が「読みやすさ」と「実行時コスト」の境界になりやすいです。

この Part では、日常のコードレビューで見つけやすい落とし穴を中心に扱います。最初から低レベル最適化へ寄せるのではなく、hot path か、入力サイズが増えるか、失敗時にどう振る舞うか、計測で差が出ているかを確認してから判断します。

## このPartで身につくこと

- allocation、boxing、LINQ、文字列処理、async、例外のコストを、必要な場面でだけ意識できる。
- 可読性を犠牲にしてよい最適化と、まず測定すべき最適化を分けられる。
- BenchmarkDotNet を使う前に、比較対象、入力サイズ、測定環境、改善幅を揃える重要性を説明できる。
- パフォーマンス指摘を、正しさ、保守性、失敗時の挙動と合わせてレビューできる。

## 読む順番

1. [allocation を意識する場面](allocation-を意識する場面.md)で、GC コストを気にするべき場所を押さえる。
2. [boxing / unboxing](boxing-unboxing.md)で、値型が `object` や interface 境界で隠れてコスト化する場面を見る。
3. [LINQ の過剰使用](linq-の過剰使用.md)と[文字列処理のコスト](文字列処理のコスト.md)で、読みやすさとループ内コストの境界を確認する。
4. [async の落とし穴](async-の落とし穴.md)で、非同期化そのものが性能改善ではないことを整理する。
5. [例外を制御フローに使わない](例外を制御フローに使わない.md)で、失敗が通常系になる処理の設計を見直す。

## 重要トピック

allocation は、すべて避けるものではありません。ループ内、高頻度 API、ログ無効時でも評価される文字列補間、大量データ処理の中間コレクションなど、GC に効きやすい場所を優先します。

boxing は、値型を `object`、非ジェネリック collection、interface、可変長引数などへ渡すと見えにくく発生します。性能が問題になる場面では、ジェネリック API、型付き comparer、struct の使い方を確認します。

LINQ は読みやすさのために有効ですが、hot path での多段列挙、`ToList()` の連発、`Count()` の重複、`GroupBy()` による大きな中間構造は注意します。async は thread を増やす仕組みではなく、待機を効率化する仕組みとして扱います。BenchmarkDotNet は結論を出す道具ではなく、仮説を検証する道具として使います。

## 実務での使いどころ

- 大量データ、バッチ処理、Web API の高頻度 endpoint、リアルタイム処理をレビューするとき。
- ログ、文字列整形、LINQ、例外処理がループや hot path に入っているとき。
- async 化したのに遅い、または thread pool 枯渇や timeout が起きているとき。
- パフォーマンス指摘に、計測結果や再現条件を添える必要があるとき。

## よくある落とし穴

- 測定せずに `Span<T>` や手書きループへ置き換え、可読性だけを落としてしまう。
- `ToList()` や `ToArray()` を確認用に挟んだまま、大量データで中間 allocation を増やす。
- 値型を使えば速いと思い込み、boxing、コピーコスト、可変 struct の扱いを見落とす。
- `async` を付ければ速くなると考え、同期 I/O、ブロッキング待機、キャンセル未対応を残す。
- 例外を通常の分岐に使い、失敗が多い入力で例外生成コストとログノイズを増やす。
- micro benchmark の改善だけで、本番の I/O、DB、ネットワーク待ちが支配的な処理まで最適化する。

## レビュー観点

- 対象コードは hot path か。入力サイズや呼び出し頻度が増える見込みはあるか。
- allocation、boxing、LINQ、文字列処理のコストが、実際に支配的になり得る場所か。
- `IEnumerable<T>` の遅延評価、複数回列挙、即時評価の境界が意図どおりか。
- async で、`CancellationToken`、timeout、例外伝播、ブロッキング待機が適切に扱われているか。
- 例外は異常系として使われ、通常の存在確認や変換失敗は `Try...` パターンなどで表現できないか。
- BenchmarkDotNet の結果は、比較対象、入力サイズ、環境、ばらつき、改善幅が説明されているか。
