# 読みやすい条件式

## 概要

条件式は、バグが入りやすくレビューでも見落としやすい場所です。複雑な条件は boolean 変数や小さなメソッドに名前を付けると、読み手が業務ルールとして確認しやすくなります。

早期 return や guard clause は、異常系や対象外ケースを先に取り除き、正常系を浅いインデントで読ませるために有効です。ただし、return が多すぎて状態の変化を追いにくい場合は、分岐の整理が必要です。

```csharp
public Discount CalculateDiscount(Customer customer, Order order)
{
    if (order.TotalAmount <= 0)
    {
        return Discount.None;
    }

    var isLoyalCustomer = customer.RegisteredAt <= clock.Today.AddYears(-3);
    var isLargeOrder = order.TotalAmount >= Money.FromYen(100_000);

    if (isLoyalCustomer && isLargeOrder)
    {
        return Discount.Percent(10);
    }

    return Discount.None;
}
```

## 判断基準

- 条件の意味が業務用語で説明できるなら、boolean 変数やメソッド名にする。
- null、空、権限なし、状態不正などは guard clause で先に返すと読みやすい。
- `&&` と `||` が混ざる条件は、括弧だけでなく名前を付けて分ける。
- 否定条件が続く場合は、肯定形の名前や早期 return に置き換えられないか見る。
- 状態の組み合わせが増えているなら、`switch` expression や状態オブジェクトを検討する。

## 使いどころ

- 条件が業務ルールそのもので、レビュー時に意味を確認したいとき。
- null、空コレクション、権限なし、状態不正などを先に除外したいとき。
- `switch` expression や pattern matching で、状態ごとの分岐を一覧しやすくしたいとき。

## 避けたい書き方

- `!` が重なった否定条件を深い `if` の中で読む必要がある。
- `&&` と `||` が長く連なり、どの条件がどの業務ルールに対応するか分からない。
- 三項演算子を入れ子にして、正常系と例外系を同じ密度で詰め込む。
- `switch` の default で未知の状態を雑に握りつぶす。
- 条件の途中で副作用のあるメソッドを呼び、評価順に依存する。

```csharp
// 避けたい例: 条件の意図をレビューしにくい。
if (!user.IsDeleted && (user.Role == Role.Admin || user.Role == Role.Owner) && !order.IsLocked)
{
    Approve(order);
}
```

```csharp
var hasApprovalRole = user.Role is Role.Admin or Role.Owner;
var canApprove = user.IsActive && hasApprovalRole && !order.IsLocked;

if (canApprove)
{
    Approve(order);
}
```

```csharp
var nextStatus = order.Status switch
{
    OrderStatus.Pending when payment.IsConfirmed => OrderStatus.Paid,
    OrderStatus.Pending => OrderStatus.AwaitingPayment,
    OrderStatus.Cancelled => OrderStatus.Cancelled,
    _ => throw new InvalidOperationException($"未対応の状態です: {order.Status}")
};
```

## レビュー観点

- 条件に名前を付けることで、業務ルールとして読めるようになっているか。
- guard clause により、主処理のインデントが深くなりすぎていないか。
- 否定条件が続く場合、肯定形に言い換えられないか。
- enum や状態分岐で、未知の値や追加された値を検知できる書き方になっているか。
- 条件式の中にログ出力、DB 更新、通知などの副作用が混ざっていないか。
- 境界値や組み合わせがテストで表現されているか。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
