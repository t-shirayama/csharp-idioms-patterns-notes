# メソッド分割、責務分割

## 概要

メソッド分割は行数を減らすためだけの作業ではありません。入力の検証、業務判断、永続化、外部呼び出し、表示用変換などの責務を分けることで、変更理由を小さくし、テストしやすくします。

長いメソッドでも、単一の流れを上から読めるなら許容できる場合があります。一方で、複数の抽象度が混ざり、途中で状態を書き換え続けるメソッドは、行数以上に保守が難しくなります。

```csharp
public async Task ApproveAsync(OrderId orderId, CancellationToken cancellationToken)
{
    var order = await orderRepository.GetAsync(orderId, cancellationToken);

    EnsureCanApprove(order);

    order.Approve(clock.UtcNow);

    await orderRepository.SaveAsync(order, cancellationToken);
}

private static void EnsureCanApprove(Order order)
{
    if (order.Status != OrderStatus.Pending)
    {
        throw new InvalidOperationException("承認待ちの注文だけ承認できます。");
    }
}
```

## 判断基準

- 同じメソッド内で「取得する」「判断する」「保存する」「通知する」が混ざる場合、境界を見直す。
- 切り出したメソッド名が本文の要約になっているなら有効。`ProcessPart1` のような名前になるなら分割単位が曖昧。
- private メソッドへの分割は、読みやすさには効くが単体テストの境界とは限らない。
- 業務ルールをテストしたいなら、値オブジェクト、ドメインサービス、ポリシークラスへ寄せる選択肢もある。
- 引数が増える場合は、呼び出し側の知識が増えているサインとして見る。

## 使いどころ

- 業務判断を単体テストしやすい形に切り出したいとき。
- 外部 I/O と純粋な計算を分け、失敗やリトライの境界を明確にしたいとき。
- メソッド内に複数のコメント見出しが必要になってきたとき。
- 引数が増えすぎて、必要なデータのまとまりが見えにくくなったとき。

## 避けたい書き方

- 何でも `private` メソッドに切り出し、呼び出し元を追わないと流れが分からない状態にする。
- `Service` や `Manager` に業務処理を集めすぎ、クラスの変更理由を広げる。
- 引数が多いメソッドを放置し、呼び出し側が順序や組み合わせを間違えやすくする。
- テストのためだけに本来隠したい詳細を public にする。
- メソッド名は抽象的なのに、本文ではログ、DB、HTTP、計算が同じ粒度で混ざっている。

```csharp
// 避けたい例: 引数の意味と組み合わせが呼び出し側に漏れている。
CreateInvoice(userId, amount, taxRate, discountRate, true, false);
```

```csharp
var request = new InvoiceRequest(
    userId,
    amount,
    taxRate,
    discountRate,
    IssueOptions.SendEmail);

CreateInvoice(request);
```

```csharp
// 避けたい例: 外部 I/O と判断が密結合で、業務ルールだけをテストしにくい。
if (await inventoryClient.GetStockAsync(productId) > 0 &&
    order.TotalAmount >= campaign.MinimumAmount)
{
    order.ApplyCampaign(campaign.Id);
}
```

```csharp
var stock = await inventoryClient.GetStockAsync(productId, cancellationToken);

if (campaignPolicy.CanApply(order, campaign, stock))
{
    order.ApplyCampaign(campaign.Id);
}
```

## レビュー観点

- メソッド名と本文の抽象度が揃っているか。
- 変更理由が異なる処理を同じクラスやメソッドに詰め込んでいないか。
- private メソッドの切り出しが、読みやすさやテストしやすさに実際に効いているか。
- 引数が多い場合、値オブジェクトや条件オブジェクトとしてまとめる方が自然ではないか。
- 外部 I/O、時刻、乱数、環境変数などの依存が、テスト可能な境界に置かれているか。
- 分割によって例外、トランザクション、キャンセルの責任が曖昧になっていないか。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
