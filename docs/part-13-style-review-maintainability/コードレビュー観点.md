# コードレビュー観点

## 概要

コードレビューは、好みの指摘を集める場ではなく、変更が安全に目的を満たしているかを確認する場です。correctness、readability、maintainability、testability、performance、security、operational readiness を分けて見ると、指摘の優先度を揃えやすくなります。

すべてを同じ重さで見る必要はありません。ユーザー入力、課金、権限、データ削除、外部連携、非同期処理のような失敗時の影響が大きい変更では、レビューの深さを上げます。

## 判断基準

- まず「この変更で守るべき仕様」を確認し、その後に実装の読みやすさを見る。
- 指摘は blocker、suggestion、question などの温度感を分ける。
- 好みの指摘は analyzer や `.editorconfig` に寄せ、人間は設計、仕様、失敗時の挙動に集中する。
- パフォーマンス指摘は hot path、入力件数、測定有無を添える。
- テスト不足の指摘は「何が壊れたら困るか」を具体的に書く。

## 使いどころ

- Pull Request の説明やレビューコメントで、指摘の種類を明確にしたいとき。
- チームで「何を見ればよいか」が属人化しているとき。
- 実装者とレビュアーが、仕様、テスト、運用影響を同じ順番で確認したいとき。

```csharp
// 例: レビューでは、null 入力、権限、監査ログ、重複実行時の挙動を確認する。
public async Task DeleteUserAsync(UserId userId, CancellationToken cancellationToken)
{
    var user = await userRepository.GetAsync(userId, cancellationToken);

    user.MarkAsDeleted(clock.UtcNow);

    await userRepository.SaveAsync(user, cancellationToken);
}
```

## 避けたい書き方

- フォーマットや命名の好みだけを指摘し、仕様上のリスクを見落とす。
- 「なんとなく読みにくい」で終わり、どの観点の問題かを伝えない。
- テストがない変更を、手元で動いたという理由だけで通す。
- パフォーマンス指摘を測定なしに広げ、可読性や正しさを下げる。
- レビュー中に別件の大きなリファクタリングを要求し、変更目的をぼかす。

```csharp
// 避けたい例: 更新と通知の順序、再実行時の挙動、失敗時の扱いが見えにくい。
public async Task ShipAsync(Order order)
{
    order.Status = OrderStatus.Shipped;
    await mailer.SendShippedAsync(order.CustomerEmail);
    await orderRepository.SaveAsync(order);
}
```

```csharp
public async Task ShipAsync(Order order, CancellationToken cancellationToken)
{
    order.MarkAsShipped(clock.UtcNow);
    await orderRepository.SaveAsync(order, cancellationToken);
    await shipmentNotifier.NotifyAsync(order.Id, cancellationToken);
}
```

## レビュー観点

- correctness: 仕様、境界値、例外、null、重複実行、並行実行に耐えるか。
- readability: 名前、分岐、抽象度が読み手に意図を伝えているか。
- maintainability: 変更理由が局所化され、不要な依存が増えていないか。
- testability: 重要な分岐を自動テストで確認でき、外部依存を分離できているか。
- performance: 測定すべき箇所と、過剰最適化を避ける箇所を区別できているか。
- security: 認可、入力検証、ログへの機密情報出力、依存ライブラリの扱いに問題がないか。
- operational readiness: ログ、メトリクス、リトライ、アラート、ロールバックを考慮しているか。
- compatibility: public API、DB schema、メッセージ形式、設定値の変更が利用者へ与える影響を確認しているか。
- observability: 失敗時に原因を追えるログや相関 ID があり、機密情報を出していないか。

## レビューコメントの書き方

- 事実、リスク、提案を分ける。
- 変更を求める場合は、なぜ必要かを仕様や運用影響に結びつける。
- 疑問は question として出し、前提が違えば取り下げられる形にする。

```text
blocker: userId が存在しない場合に削除済みとして成功扱いになります。
監査ログ上は削除されたように見えるため、404 か domain error として返す方が安全そうです。
```

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
