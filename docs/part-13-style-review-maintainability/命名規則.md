# 命名規則

## 概要

命名は、コード上で最も頻繁に読まれる設計情報です。クラス名は責務、メソッド名は操作、変数名は役割、boolean 名は真偽の意味を表します。C# では .NET の一般的な命名規則に合わせ、public な型やメンバーは `PascalCase`、ローカル変数や引数は `camelCase` にするのが基本です。

よい名前は「実装の都合」より「呼び出し側が知るべき概念」を表します。`SqlUserRepository` のように技術を表す名前が有効な境界もありますが、ドメインの中心では `UserRegistrationService` や `InvoicePolicy` のように、変更理由が名前から分かる方が保守しやすくなります。

`Async` 接尾辞、`Try` パターン、`Is` / `Has` / `Can` などの boolean 接頭辞は、呼び出し側の期待値を作ります。名前だけで戻り値、失敗時の扱い、待機の必要性がある程度分かることが重要です。

```csharp
public async Task<User?> FindUserAsync(
    string email,
    CancellationToken cancellationToken)
{
    return await userRepository.FindByEmailAsync(email, cancellationToken);
}

public bool CanApprove(Order order)
{
    return order.TotalAmount <= approvalLimit &&
           order.Status == OrderStatus.Pending;
}
```

## 判断基準

- 型名は「何を持っているか」ではなく「何の責務を持つか」で付ける。
- メソッド名は副作用の有無が分かる動詞にする。`Get`、`Find`、`Create`、`Save` は失敗時の扱いも含めて揃える。
- boolean は `true` の意味を肯定形で読める名前にする。`isNotInvalid` のような二重否定は避ける。
- DTO、Entity、Value Object、Service などの接尾辞は、チーム内で意味が共有されている場合だけ使う。
- 略語は `Id`、`Uri`、`Db` など一般的なものに寄せ、同じ概念に複数表記を混ぜない。

## 使いどころ

- ドメイン上の概念を型名やメソッド名に反映し、実装詳細の名前を外へ漏らしたくないとき。
- 非同期処理、失敗しうる変換、条件判定など、呼び出し側が扱いを間違えやすい API。
- レビューで「何をしているか」より「それでよいか」を議論したいコード。

## 避けたい書き方

- `data`、`item`、`manager`、`helper` のように、役割が広すぎる名前を増やす。
- boolean に `flag` や `check` だけを付け、`true` の意味が読めない。
- 非同期メソッドに `Async` を付けず、呼び出し側が待機や例外の扱いを見落とす。
- 略語をチーム内で揃えず、`Customer` / `Cust` / `Cstmr` のように揺らす。
- 現在の実装だけを名前に入れ、将来の変更で名前だけ古くなる。

```csharp
// 避けたい例: true の意味も、何を処理するのかも読み取りにくい。
var flag = Check(data);

if (flag)
{
    DoWork(data);
}
```

```csharp
var canSendReminder = invoice.IsOverdue &&
                      invoice.Customer.AcceptsEmail;

if (canSendReminder)
{
    SendReminder(invoice);
}
```

## レビュー観点

- 名前から責務、戻り値、失敗時の扱いが推測できるか。
- boolean 名は `true` のときに何が成り立つかを自然に読めるか。
- 略語は一般的なものに限定され、同じ概念に複数の表記が混ざっていないか。
- 型名やメソッド名が、現在の実装ではなくドメイン上の意図を表しているか。
- `Get` と `Find`、`Create` と `Build` など、似た名前の使い分けがチーム内で一貫しているか。
- 名前の曖昧さをコメントで補っていないか。名前や型で表現できるならそちらを優先する。

## TODO対応: 単位付き命名

> 対応元: P1 / 単位付き命名、nullable を示す名前、collection の単数複数、`Try` / `Find` / `Get` の失敗時契約、`CancellationToken` 付き async API の命名例を追加する。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
