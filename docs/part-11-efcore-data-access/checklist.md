# EF Core / DB アクセス レビュー checklist

EF Core のレビューでは、C# のコードだけでなく、実際に発行される SQL、データ量、トランザクション境界、スキーマ変更、運用時の失敗を確認します。

## クエリ設計

- [ ] 必要な列だけを projection しているか
- [ ] 一覧や検索で `OrderBy`、`Skip`、`Take` などの件数制御が明示されているか
- [ ] 読み取り専用クエリに `AsNoTracking()` を使うかどうか判断しているか
- [ ] `ToListAsync()` などの即時評価位置が適切か
- [ ] `IEnumerable<T>` 化した後に DB で処理すべき絞り込みをしていないか

## Include と N+1

- [ ] `Include` で読み込むナビゲーションがレスポンスや処理に本当に必要か
- [ ] ループ内で追加クエリが発生しないか
- [ ] join が巨大化する場合に projection や split query を検討しているか
- [ ] lazy loading を使う場合、発行 SQL を追跡できる状態になっているか
- [ ] テストデータが少なすぎて N+1 を見落としていないか

## 更新と整合性

- [ ] 更新対象 Entity の取得、変更、保存の流れが明確か
- [ ] 複数テーブル更新のトランザクション境界が適切か
- [ ] 外部 I/O を DB トランザクション内に入れていないか
- [ ] 同時更新時の挙動を考慮しているか
- [ ] `DbUpdateConcurrencyException` の扱いが業務要件に合っているか

## Migration

- [ ] migration の Up / Down をレビューしているか
- [ ] 既存データがある状態で適用できるか
- [ ] nullable 変更、default 値、データ移行が明示されているか
- [ ] index 作成や列追加で長時間ロックが起きないか
- [ ] アプリケーションのデプロイ順序と互換性が考慮されているか

## Repository と境界

- [ ] Repository が EF Core の薄いラッパーだけになっていないか
- [ ] `IQueryable<T>` を公開する場合、責務境界が崩れていないか
- [ ] UseCase / Service が業務判断を持ち、Repository が永続化詳細に閉じているか
- [ ] テストのためだけに過剰な抽象化を追加していないか
- [ ] EF Core の機能を隠しすぎて、性能や SQL の確認が難しくなっていないか

## Raw SQL とセキュリティ

- [ ] 値はパラメータ化されているか
- [ ] テーブル名、列名、並び順などの識別子は whitelist で制限しているか
- [ ] ユーザー入力を文字列連結で SQL に埋め込んでいないか
- [ ] raw SQL を使う理由が LINQ では表現しにくい処理や性能上の必要性として説明できるか
- [ ] raw SQL の結果型、権限、監査ログ、テストが確認されているか

## テストと観測性

- [ ] SQLite や実 DB に近い provider で重要クエリを確認しているか
- [ ] InMemory provider だけで SQL 変換、制約、transaction を検証したつもりになっていないか
- [ ] SQL ログ、実行時間、件数、相関 ID を追えるか
- [ ] 大量データ、ゼロ件、重複、同時更新、制約違反のテストがあるか
- [ ] 性能問題の指摘に、SQL、実行計画、データ量、測定条件が添えられているか
