# ログと構造化ログ

## 概要

ログは障害時に「何が起きたか」を再構成するための情報です。`ILogger<T>` を使い、文字列連結ではなく構造化ログとして値を渡すと、検索や集計がしやすくなります。

ログレベルは読み手への合図です。通常の業務分岐を `Error` にしすぎると監視が鈍り、本当に対応が必要な失敗が埋もれます。一方で、例外をログに残さず上位へ投げるだけにすると、境界で文脈が失われることがあります。

```csharp
logger.LogInformation(
    "Order {OrderId} was submitted by user {UserId}",
    order.Id,
    userId);

try
{
    await paymentClient.ChargeAsync(order.Id, cancellationToken);
}
catch (PaymentTimeoutException ex)
{
    logger.LogWarning(ex, "Payment timed out for order {OrderId}", order.Id);
    throw;
}
```

## 使いどころ

- API、バッチ、メッセージ処理など、後から実行経路を追いたい境界。
- 外部サービス呼び出し、リトライ、タイムアウト、ユーザー操作の主要イベント。
- 監視やアラートの条件に使う失敗や状態変化。

## 判断基準

ログは「多いほど安心」ではありません。読み手が障害調査で使える情報か、監視で意味を持つレベルか、個人情報を残していないかを基準にします。

- `Trace` / `Debug`: 開発時や一時的な詳細調査に使う。
- `Information`: 正常な業務イベントや処理の節目を残す。
- `Warning`: リトライ可能な失敗、業務上の否認、期待外だが継続できる状態を残す。
- `Error`: 処理が失敗し、運用上の対応や調査が必要なものを残す。
- `Critical`: プロセス継続やシステム全体に影響する重大障害に限定する。

例えば、ログイン失敗は通常のユーザー操作として `Information` または `Warning` を検討します。全件を `Error` にすると、攻撃検知や障害検知のノイズになりやすいです。

## 構造化ログの書き方

メッセージテンプレートは、後から検索・集計したい値を名前付きで渡します。テンプレート名は揺らさず、同じ概念には同じプロパティ名を使います。

```csharp
logger.LogInformation(
    "Payment authorized for order {OrderId} with provider {Provider}",
    orderId,
    providerName);
```

`orderId` をログに残すなら、API の相関 ID、ジョブ ID、外部サービスの request ID なども合わせて残すと、複数システムをまたぐ調査がしやすくなります。

```csharp
using var scope = logger.BeginScope(new Dictionary<string, object>
{
    ["CorrelationId"] = correlationId,
    ["UserId"] = userId
});

logger.LogInformation("Creating order {OrderId}", orderId);
```

scope は便利ですが、全ログに個人情報を混ぜると影響範囲が大きくなります。ユーザーを識別する必要がある場合も、メールアドレスなどではなく内部 ID やハッシュ化された値を検討します。

## 避けたい書き方

- `$"user {userId} failed"` のように文字列補間で値を埋め込み、構造化検索できなくする。
- 個人情報、アクセストークン、クレジットカード番号などをログに出す。
- 例外を `LogError` した直後に別の例外で包み直し、同じ失敗を何度も Error として出す。
- ループ内で大量の `Information` を出し、正常処理のたびにログ基盤の負荷を上げる。
- 例外を握りつぶすための言い訳としてログだけ残す。

```csharp
// 避けたい例: Email がログ基盤に残り、検索キーも構造化されない。
logger.LogInformation($"Login failed: {request.Email}");
```

```csharp
// 避けたい例: ログは残るが、呼び出し元から失敗が見えなくなる。
catch (Exception ex)
{
    logger.LogError(ex, "Failed to import file {FileName}", fileName);
    return ImportResult.Success();
}
```

## 例外ログの境界

例外は、捕捉したすべての場所で `Error` にするのではなく、責務の境界で一度だけ記録する方が読みやすいです。下位層では例外に文脈を足して再送出し、API middleware や worker のトップレベルでログと応答に変換する、という分け方がよく使われます。

```csharp
catch (HttpRequestException ex)
{
    throw new PaymentGatewayException(orderId, "Payment provider failed.", ex);
}
```

外部依存の失敗では、相手先名、操作名、対象 ID、再試行したかどうかを残します。一方で request / response body を丸ごと出すと、secret や個人情報を含みやすいため、出す場合は明示的にマスクします。

## レビュー観点

- ログメッセージがテンプレートと引数で書かれているか。
- `Trace` / `Debug` / `Information` / `Warning` / `Error` の使い分けが運用上の意味と合っているか。
- 個人情報や secret が含まれていないか。
- 失敗時に調査に必要な ID、相関 ID、外部依存名が残るか。
- 同じ失敗が複数箇所で Error として重複記録されていないか。
- 正常系ログの量が、運用コストや検索性を悪化させないか。
- ログから復旧判断に必要な情報は分かるが、内部実装を過剰に露出していないか。
