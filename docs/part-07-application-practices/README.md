# Part 7: 実務アプリケーションの定石

アプリケーションコードでは、文法の正しさだけでなく、設定、ログ、入力検証、エラー、層の責務、API 境界、DB アクセスの扱いが保守性を大きく左右します。

この Part では、ASP.NET Core などの現代的な C# アプリケーションでよく出る判断を、レビューで確認しやすい単位に整理します。重要なのは「どこに書けるか」ではなく、「どの責務をどこに置くと、変更や障害調査に強くなるか」です。

## このPartで身につくこと

- 環境差分を安全に扱う設定と Options pattern の使い方
- 障害調査で役に立つ構造化ログの残し方
- 入力検証、業務ルール、永続化制約を混ぜない判断
- 例外を握りつぶさず、利用者と運用者に必要な情報へ変換する設計
- Repository / Service / UseCase の責務分離と、分けすぎない基準
- Web API の DTO、境界、マッピングで内部モデルを漏らさない考え方
- DB アクセスで性能、整合性、トランザクション、N+1 をレビューする観点

## 重要トピック

- 設定値は文字列の直読みではなく、型付き Options と検証で扱う
- ログは文章ではなく、検索できるイベントとして設計する
- Validation は入力形式、業務制約、DB 制約のどれを守っているかを分けて考える
- 例外は内部原因、HTTP レスポンス、利用者向けメッセージを混同しない
- Service や UseCase は「何でも置き場」ではなく、業務判断の単位で切る
- DTO と Entity を安易に共有せず、API 契約と内部都合を分離する
- DB ではデータ量、遅延読み込み、トランザクション境界、同時更新を確認する

## 実務での使いどころ

新規機能の API を追加するときは、リクエスト DTO、Validation、UseCase、Repository、レスポンス DTO、ログ、エラー応答を一つの流れとして確認します。どこか一箇所だけがきれいでも、境界で責務が混ざると変更時に崩れやすくなります。

障害対応や運用改善では、ログの粒度、相関 ID、例外の分類、設定の反映方法が効いてきます。特に本番環境だけで起きる問題は、設定と外部 I/O と DB の境界が曖昧なコードほど追いにくくなります。

## よくある落とし穴

- `IConfiguration` から各所で直接値を読む
- ログに必要な ID がなく、後から処理を追跡できない
- Validation に業務判断、認可、DB 存在確認が詰め込まれている
- 例外を catch してログだけ出し、呼び出し側には成功のように見せる
- Service 層が巨大化し、UseCase と Repository と DTO 変換が混ざる
- Entity をそのまま API レスポンスに使い、内部構造や余計な項目を公開する
- 小さいデータでしか確認せず、N+1 や過剰なメモリ展開に気づかない

## レビュー観点

- 設定値に既定値、必須チェック、環境別の差分管理があるか
- ログに操作対象 ID、ユーザーやテナント、相関 ID、失敗理由が構造化されているか
- Validation の責務が入力形式、業務ルール、永続化制約で混ざっていないか
- 例外の分類、再試行可否、HTTP ステータス、利用者向け文言が適切か
- UseCase が業務判断を表し、Repository が永続化の詳細に閉じているか
- DTO マッピングで内部モデル、機密項目、余計なナビゲーションが漏れていないか
- DB クエリがデータ量、インデックス、トランザクション、同時更新に耐えられるか
