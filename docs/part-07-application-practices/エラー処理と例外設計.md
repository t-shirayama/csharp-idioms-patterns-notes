# エラー処理と例外設計

## 概要

例外は「通常の分岐ではなく、呼び出し元がその場で処理を継続しにくい失敗」を表すために使います。入力ミスや業務上よく起こる否認をすべて例外にすると、制御フローと監視の両方が読みづらくなります。

一方で、外部依存の障害、整合性違反、プログラム上の前提崩れは例外として扱う方が自然です。業務上の成功/失敗を値で返したい場合は、Result 型的な戻り値を使うと呼び出し側の判断が明確になります。

```csharp
public sealed record Result<T>(T? Value, string? ErrorCode)
{
    public bool IsSuccess => ErrorCode is null;

    public static Result<T> Success(T value) => new(value, null);
    public static Result<T> Failure(string errorCode) => new(default, errorCode);
}

public Result<Order> Cancel(Order order)
{
    if (order.Status is OrderStatus.Shipped)
    {
        return Result<Order>.Failure("ORDER_ALREADY_SHIPPED");
    }

    return Result<Order>.Success(order.Cancel());
}
```

## 使いどころ

- DB 接続失敗、外部 API 障害、想定外の状態など、処理継続が難しい失敗は例外にする。
- 在庫不足、権限不足、状態遷移不可など、業務上想定される否認は戻り値で表す。
- 境界で例外を捕捉し、HTTP ステータス、ログ、監視に変換する。

## 例外にするか値で返すか

迷ったときは、呼び出し元が通常の分岐として扱うべきかを見ます。

- ユーザーの入力修正で解決する: validation error。
- 業務上よく起こる否認: Result やエラーコード。
- 外部依存やインフラ障害: 例外。
- プログラムの前提違反: 例外。
- キャンセル: `OperationCanceledException` / `TaskCanceledException` を尊重する。

```csharp
public async Task<Result<bool>> ReserveAsync(
    Guid itemId,
    int quantity,
    CancellationToken cancellationToken)
{
    var stock = await stockRepository.FindAsync(itemId, cancellationToken);
    if (stock is null)
    {
        return Result<bool>.Failure("ITEM_NOT_FOUND");
    }

    if (!stock.CanReserve(quantity))
    {
        return Result<bool>.Failure("INSUFFICIENT_STOCK");
    }

    stock.Reserve(quantity);
    await stockRepository.SaveAsync(stock, cancellationToken);
    return Result<bool>.Success(true);
}
```

Result 型を使う場合も、すべての失敗を値に押し込む必要はありません。DB 接続失敗まで Result にすると、呼び出し側が回復できないエラーを毎回分岐することになり、逆に見通しが悪くなります。

## 境界で変換する

Web API では、例外や Result を Controller の各 action でばらばらに変換すると、ステータスコードやレスポンス形式が揺れます。例外 middleware、filter、endpoint filter など、境界で一貫して変換する場所を作ります。

```csharp
catch (OrderNotFoundException ex)
{
    return Results.Problem(
        statusCode: StatusCodes.Status404NotFound,
        title: "Order was not found.",
        extensions: new Dictionary<string, object?>
        {
            ["code"] = "ORDER_NOT_FOUND",
            ["orderId"] = ex.OrderId
        });
}
```

利用者向けメッセージとログ向けメッセージは分けます。レスポンスには内部実装や接続先の詳細を出さず、ログには調査に必要な ID と文脈を残します。

## 例外型の設計

独自例外は、呼び出し側が区別して扱う、またはログ・監視で分類したい場合に意味があります。単に名前を変えるだけなら、標準例外とメッセージで十分なこともあります。

```csharp
public sealed class OrderNotFoundException : Exception
{
    public Guid OrderId { get; }

    public OrderNotFoundException(Guid orderId)
        : base($"Order was not found. OrderId: {orderId}")
    {
        OrderId = orderId;
    }
}
```

例外に値を持たせると、文字列解析に頼らずログや応答へ変換できます。ただし、secret や個人情報を例外プロパティに持たせると、ログに流れやすくなるため避けます。

## 避けたい書き方

- `catch (Exception) { }` で握りつぶし、成功したように処理を続ける。
- 例外メッセージの文字列比較で分岐する。
- 独自例外を大量に作るが、呼び出し側で区別していない。
- `OperationCanceledException` を通常のエラーとしてログに積み、キャンセルを障害扱いする。
- 例外を包み直すたびに元の例外を `InnerException` として残さない。

```csharp
// 避けたい例: 失敗の事実も原因も失われる。
try
{
    await importer.ImportAsync(cancellationToken);
}
catch (Exception)
{
}
```

```csharp
// 避けたい例: 文字列比較はローカライズや文言変更に弱い。
if (ex.Message.Contains("not found"))
{
    return NotFound();
}
```

## レビュー観点

- 例外にする失敗と戻り値にする失敗の基準が一貫しているか。
- 例外を捕捉した場所で、ログ、再送出、利用者向け応答の責務が明確か。
- 監視したい失敗に識別できるコードや例外型があるか。
- エラーメッセージに secret や内部実装の詳細を出していないか。
- `InnerException`、対象 ID、外部依存名など、調査に必要な文脈が残るか。
- キャンセル、タイムアウト、リトライ不能な失敗を混同していないか。
- HTTP ステータスやエラーコードへの変換がアプリ全体で揺れていないか。
