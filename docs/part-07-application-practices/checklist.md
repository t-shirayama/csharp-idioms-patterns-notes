# Part 7: 実務アプリケーションのレビュー checklist

このチェックリストは、ASP.NET Core などの C# アプリケーションをコードレビューするときに、設定、ログ、API 境界、DB アクセス、エラー処理の観点を抜け漏れなく確認するためのものです。

すべての項目を毎回機械的に満たすための表ではありません。PR の変更点に関係する章だけを見て、「本番データ量」「障害時」「設定差分」「利用者に返るエラー」を具体的に想像するために使います。

## 設定と Options

- [ ] 環境ごとに変わる値、機能フラグ、タイムアウト、リトライ回数が、コード中の固定値ではなく設定として外に出ている。
- [ ] `IOptions<T>`、`IOptionsSnapshot<T>`、`IOptionsMonitor<T>` の使い分けが、DI のライフタイム、再読み込み要件、スレッド安全性に合っている。
- [ ] 起動時に `ValidateOnStart` などで必須設定、範囲、URL 形式、相互依存を検証し、深い処理で初めて設定漏れが発覚しない。
- [ ] named options や section 名が用途単位で分かれており、似た設定を別サービスへ誤って注入しない。
- [ ] 接続文字列、API key、token などの secret がログ、例外メッセージ、設定サンプル、リポジトリに露出していない。
- [ ] 本番、検証、ローカルで既定値が変わる設定について、危険な default が本番に流れない。
- [ ] 設定クラスは用途ごとに分かれており、巨大な `AppSettings` に依存が集中していない。

## ログと監視

- [ ] ログは文字列連結や補間ではなく、検索しやすい構造化ログのテンプレートになっている。
- [ ] `Information`、`Warning`、`Error` などのレベルが、アラート対象、調査対象、利用者操作ログの違いと合っている。
- [ ] request id、correlation id、tenant id、user id、外部システムの識別子など、障害調査に必要な文脈が scope として残る。
- [ ] 外部 API 呼び出しや DB 更新では、対象、結果、所要時間、再試行回数など、後から絞り込める最小限の情報がある。
- [ ] 高頻度ループや大量データ処理で、ログ量、ラベルの cardinality、課金や保存期間への影響を確認している。
- [ ] 例外ログは境界で一度だけ記録し、同じ例外を複数層で重複して出していない。
- [ ] 個人情報、認証情報、入力値の全文、カード番号、認可ヘッダーなど、残すべきでない値をログに出していない。

## Validation とエラー処理

- [ ] 入力形式の検証と業務ルールの検証が混ざりすぎていない。
- [ ] 利用者に修正してほしいエラーと、システム側で調査すべき例外を区別している。
- [ ] API のエラーレスポンスは、HTTP status、エラーコード、利用者向けメッセージ、調査用 trace id の責務が明確になっている。
- [ ] model binding 失敗、認証失敗、認可失敗、存在しないリソース、競合、重複登録を別の失敗として扱っている。
- [ ] 例外を通常の分岐制御に使っていない。想定内の失敗は値や結果型で表現できないか検討している。
- [ ] `catch (Exception)` で握りつぶさず、再送、補足情報の付与、境界での変換など目的が明確になっている。
- [ ] 例外を再 throw するときに `throw ex;` で stack trace を壊していない。
- [ ] 外部サービスや DB の詳細な例外を、そのまま API 利用者へ返していない。

## Web API と DTO 境界

- [ ] 外部公開する DTO と内部ドメインモデル、Entity Framework の entity が不用意に共有されていない。
- [ ] request DTO と response DTO が用途ごとに分かれており、更新できてはいけない項目を受け取らない。
- [ ] PATCH、PUT、POST の意味が揃っており、未指定、null、空文字、空配列の扱いを DTO と validation で区別できる。
- [ ] mapping の責務が controller に肥大化せず、読みやすい場所にまとまっている。
- [ ] `CreatedAtAction`、Location header、pagination、並び順など、クライアントが次に使う契約が不足していない。
- [ ] 冪等性が必要な更新や外部連携で、リトライ時に二重登録や二重実行にならない。
- [ ] 互換性を壊す変更が必要な場合、versioning や段階的移行の方針を確認している。

## 層の責務と依存

- [ ] controller、use case、service、repository の責務が説明できる単位で分かれている。
- [ ] 業務判断が controller や repository に散らばらず、テストしやすい層に置かれている。
- [ ] repository は永続化の詳細を隠しつつ、単なる CRUD ラッパーとして増えすぎていない。
- [ ] トランザクション境界が use case の一連の業務処理と合っている。
- [ ] 外部 API 呼び出し、メール送信、ファイル I/O などを DB トランザクション内に長く抱えていない。
- [ ] ドメイン判断とフレームワーク依存の変換が混ざらず、単体テストできる場所に業務ルールが残っている。
- [ ] `CancellationToken`、タイムアウト、リトライ方針が controller から外部 I/O まで途切れず伝わっている。

## DB アクセス

- [ ] N+1、不要な全件取得、巨大な `Include` など、データ量が増えたときの失敗を確認している。
- [ ] `AsNoTracking`、projection、pagination など、読み取り用途に合うクエリ形になっている。
- [ ] `Skip` / `Take` を使う場合、安定した `OrderBy` と最大件数の上限がある。
- [ ] 検索条件、外部キー、ソートキーに必要な index があり、ローカルの少量データだけで判断していない。
- [ ] 同時更新が問題になる箇所で、楽観ロック、排他、再試行、失敗時の表示方針を検討している。
- [ ] `SaveChanges` の呼び出し回数と transaction 範囲が、業務上の一貫性と性能に合っている。
- [ ] migration は意図した schema 変更だけを含み、既存データ、lock、実行時間、rollback 方針が確認されている。
- [ ] SQL や LINQ の条件にユーザー入力を使う場合、インジェクション、意図しない広範囲更新、tenant 境界の漏れを避けている。
- [ ] raw SQL や `ExecuteUpdate` / `ExecuteDelete` を使う場合、where 条件、監査項目、影響件数の確認がある。

## API・DB・例外をまたぐ確認

- [ ] API で成功に見えるが DB は一部だけ更新された、という中途半端な状態にならない。
- [ ] DB の一意制約や外部キー制約に依存する失敗を、利用者が理解できるエラーへ変換している。
- [ ] 外部 API のタイムアウトや一時障害が、無制限リトライ、長時間待機、重複実行につながらない。
- [ ] 監査ログ、更新者、更新日時、論理削除の扱いが、通常更新とバッチ更新で揃っている。

## レビューで止めて確認したい兆候

- [ ] 「とりあえずログ」「とりあえず catch」「とりあえず repository」のように、目的が説明されていない。
- [ ] 変更対象より広い層に修正が波及しており、責務の境界が崩れている。
- [ ] ローカルや単一ユーザーでは動くが、本番のデータ量、同時実行、障害時の挙動が見えていない。
- [ ] テストではなく実環境設定や外部サービスに強く依存し、再現性のある確認が難しい。
