# Part 6 レビュー用チェックリスト

オブジェクト指向、設計パターン、DI を含む変更をレビューするときは、型やパターンの名前よりも、変更理由、依存の向き、状態の守り方、テストで押さえられる境界を確認する。

## 変更理由と責務

- [ ] 1 つのクラスやメソッドに、業務判断、永続化、外部 I/O、表示用変換など異なる変更理由が混ざっていない。
- [ ] 新しい要件が来たとき、どの型を変更するか説明できる。
- [ ] パターンや抽象化を入れた結果、呼び出し側の意図が読みやすくなっている。
- [ ] 「将来使うかもしれない」だけの抽象化で、型やファイルが増えすぎていない。
- [ ] クラス名やメソッド名が、パターン名ではなく業務上の役割を表している。
- [ ] 条件分岐を移動しただけで、変更理由や依存の向きが分かりにくくなっていない。
- [ ] controller、handler、service、repository などの名前が、実際の責務の広さと合っている。

## カプセル化と状態

- [ ] 外部から変更できるプロパティ、コレクション、フィールドは、本当に公開してよい理由がある。
- [ ] 不変条件はコンストラクタ、Factory、メソッド内で守られ、不正な状態を後から検知するだけになっていない。
- [ ] コレクションを返す場合、追加、削除、並べ替えの権限を誰が持つか決まっている。
- [ ] `record`、`class`、`struct` の選択が、等価性、可変性、サイズ、用途に合っている。
- [ ] `public set` や可変 DTO をドメインモデルまで持ち込んでいない。

## 継承、interface、abstract class

- [ ] 継承を使う理由が「共通処理を置きたい」だけではなく、派生型が同じ契約で扱えることを説明できる。
- [ ] 委譲や合成で十分な変更に対して、深い継承階層を作っていない。
- [ ] `virtual` / `override` の前提条件、呼び出し順序、不変条件がテストや名前から追える。
- [ ] `interface` は呼び出し側の必要な操作を表し、実装側の都合を漏らしていない。
- [ ] `abstract class` を選ぶ場合、共通状態やテンプレート処理を持つ理由がある。
- [ ] 公開済み `interface` へメンバーを追加するとき、既存実装への影響を確認している。
- [ ] 継承させる意図がない型は `sealed` にできないか検討している。
- [ ] テスト用モックのためだけに、実装が 1 つしかない不自然な `interface` を公開 API にしていない。
- [ ] default interface method を使う場合、既存実装への互換性以外の理由で契約を曖昧にしていない。

## パターン適用

- [ ] Strategy は、条件分岐の差し替え単位が明確で、呼び出し側が分岐の詳細を知らなくてよい。
- [ ] Factory は生成責務に収まり、生成後の業務処理まで持っていない。
- [ ] Template Method は、派生型が変更してよい箇所と固定すべき手順が分かれている。
- [ ] パターン導入前より、単体テストで確認する対象が明確になっている。
- [ ] 条件追加時に、既存コードの修正が局所的で済む。
- [ ] Factory が DI コンテナや `IServiceProvider` の薄いラッパーになり、service locator 化していない。
- [ ] Strategy や Factory の選択条件が、呼び出し元の `switch` / `if` に残り続けていない。
- [ ] Decorator や pipeline を使う場合、実行順序、例外時の扱い、副作用の重複が確認されている。

## DI と依存関係

- [ ] コンストラクタを見るだけで、その型が必要とする依存を把握できる。
- [ ] `IServiceProvider`、static singleton、環境変数の直接参照で依存が隠れていない。
- [ ] DI コンテナの登録ライフタイムが、依存先の状態やスレッド安全性と一致している。
- [ ] scoped service を singleton へ注入するなど、ライフタイムの逆転がない。
- [ ] logger、options、repository、client などの依存が増えすぎて、クラスの責務が膨らんでいない。
- [ ] テストで差し替えるべき外部依存と、単なる値オブジェクトや設定値を混同していない。
- [ ] `DbContext`、request context、user context など scoped な状態を singleton や static cache に保持していない。
- [ ] `HttpClient`、DB 接続、ファイル、timer など dispose が必要な依存を、所有者が分からない形で生成していない。
- [ ] options は `IOptions<T>` / `IOptionsSnapshot<T>` / `IOptionsMonitor<T>` の違いを、更新頻度とライフタイムに合わせて選んでいる。
- [ ] 登録漏れやライフタイム不整合を、起動時検証や小さな統合テストで検出できる。
- [ ] 循環依存を解消するために lazy、factory、service locator へ逃げる前に、責務分割を見直している。
- [ ] メソッド単位で変わる値をコンストラクタ DI に入れず、引数や Value Object として渡している。

## Entity、Value Object、DTO

- [ ] その型を同一性で見るのか、値の等価性で見るのかが明確になっている。
- [ ] Value Object は生成時に不正な値を防ぎ、後続処理へ検証済みの値を渡している。
- [ ] DTO とドメインモデルを直接共有して、外部 API や DB の仕様が内部設計を縛っていない。
- [ ] mapping が冗長に見える場合でも、境界を消してよい理由を説明できる。
- [ ] Entity の状態変更は、業務ルールを通るメソッドで行われている。

## テストしやすさ

- [ ] 外部 I/O、時刻、乱数、設定、DB など、テストで制御したい依存が境界として分離されている。
- [ ] モックしやすさだけを目的に、不自然な `interface` を増やしていない。
- [ ] ドメインルールは DI コンテナやフレームワークを起動しなくてもテストできる。
- [ ] 抽象化によって、正常系だけでなく失敗系や境界値もテストしやすくなっている。
- [ ] DI 登録のテストと、ドメインルールの単体テストを混ぜて、失敗理由が読みにくくなっていない。
- [ ] テストダブルが実装詳細を固定しすぎて、リファクタリングで壊れやすいテストになっていない。

## レビューコメントの観点

- [ ] 「このパターンを使うべき」ではなく、「変更理由」「依存の向き」「テスト境界」のどれに効く指摘かを明確にしている。
- [ ] 過剰設計を疑う場合、将来性ではなく現在の変更頻度、責務の混在、テスト負荷を根拠にしている。
- [ ] シンプルな実装を提案する場合、後から拡張するときにどこを変えるかも示している。
- [ ] DI の指摘では、登録方法だけでなく、ライフタイム、所有権、状態共有、テスト境界のどれが問題かを分けて書いている。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
