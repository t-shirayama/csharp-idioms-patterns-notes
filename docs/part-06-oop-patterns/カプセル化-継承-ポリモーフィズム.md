# カプセル化、継承、ポリモーフィズム

## 概要

カプセル化は、オブジェクトの内部状態を直接触らせず、守るべき不変条件をメソッドやプロパティの境界に閉じ込める考え方です。継承とポリモーフィズムは、共通の契約で異なる実装を扱うために使いますが、変更範囲を広げやすい道具でもあります。

## 使いどころ

- 状態変更に業務ルールがあり、単なる `set` では不変条件を守れない。
- 呼び出し側が具象型を知らなくても、同じ操作として扱える。
- 派生型が本当に基底型として自然に振る舞える。
- 変更可能な値と読み取り専用の値を分け、オブジェクトの寿命中に守る条件を明確にしたい。

```csharp
public sealed class BankAccount
{
    public decimal Balance { get; private set; }

    public void Withdraw(decimal amount)
    {
        if (amount <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(amount));
        }

        if (Balance < amount)
        {
            throw new InvalidOperationException("残高が不足しています。");
        }

        Balance -= amount;
    }
}
```

## 判断基準

カプセル化は「フィールドを private にする」だけでは足りません。外から見える操作が、業務上許される状態遷移だけを表しているかを確認します。例えば残高は公開してよくても、残高を任意値に書き換える操作は公開しない、という切り分けです。

継承は、派生型が基底型の契約を自然に満たせる場合に限ると扱いやすくなります。共通コードを再利用したいだけなら、private なヘルパー、委譲、Strategy のほうが変更範囲を狭くできます。

```csharp
public sealed class Order
{
    private readonly List<OrderLine> _lines = [];

    public IReadOnlyCollection<OrderLine> Lines => _lines;

    public void AddLine(Product product, int quantity)
    {
        if (quantity <= 0)
        {
            throw new ArgumentOutOfRangeException(nameof(quantity));
        }

        _lines.Add(new OrderLine(product.Id, quantity, product.Price));
    }
}
```

## 避けたい書き方

- `public set` でどこからでも状態を変更でき、不正な状態を作れる。
- 「共通処理を置きたい」だけで継承を使い、派生型同士が暗黙の制約で結び付く。
- `virtual` を広く公開し、どのメソッドが上書き前提なのか分からない。
- `IReadOnlyList<T>` を返していても、内部の `List<T>` をそのまま外へ渡して変更される。

```csharp
public class BankAccount
{
    public decimal Balance { get; set; }
}
```

```csharp
// 避けたい例: 呼び出し側が明細を自由に差し替えられる。
public sealed class Order
{
    public List<OrderLine> Lines { get; } = [];
}
```

## よくある失敗

- setter を閉じたが、`UpdateStatus(string status)` のように何でも受け取るメソッドを作り、不正状態を防げていない。
- `protected` メンバーを増やしすぎて、派生型が基底クラスの内部実装に依存する。
- ポリモーフィズムで分岐を消したつもりが、呼び出し側に `if (x is ConcreteType)` が残っている。
- 継承階層を深くした結果、テストでどのメソッドが実際に呼ばれるか追いにくい。

## レビュー観点

- 外部から変更できる状態は、本当に公開してよいか。
- `virtual` / `override` の契約がコメントやテストで説明できるか。
- 継承より、委譲や合成で済む変更ではないか。
- `sealed` にできるクラスを開いたままにしていないか。
- コレクションを公開する場合、追加、削除、並べ替えの権限を誰が持つか決まっているか。
