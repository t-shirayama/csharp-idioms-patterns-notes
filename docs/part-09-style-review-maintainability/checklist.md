# Part 9 チェックリスト: スタイル、レビュー、保守性

このチェックリストは、スタイルや保守性のレビューで「好みの指摘」と「変更リスク」を分けるために使います。すべてを毎回見る必要はありませんが、public API、外部入力、長く保守するドメインロジックでは確認の深さを上げます。

## レビュー前の前提

- [ ] PR の目的、変更範囲、仕様上の前提が説明されているか。
- [ ] 差分の中心が、仕様変更、リファクタリング、警告対応、フォーマット変更のどれかに分かれているか。
- [ ] レビューで必ず見る箇所と、suggestion に留める箇所を分けられるか。
- [ ] 既存の設計や命名から外れる変更に、理由や移行方針があるか。

## 命名と意図

- [ ] 型名、メソッド名、変数名から、何を表す値か、どの単位か、どの状態かを読み取れるか。
- [ ] `count`、`total`、`amount`、`price` などが、件数、金額、税込/税抜、通貨、期間を曖昧にしていないか。
- [ ] boolean 名は `is`、`has`、`can` などで肯定条件を表し、`isNot...` と否定条件の二重否定を増やしていないか。
- [ ] `data`、`info`、`manager`、`helper`、`util` のような広すぎる名前が、責務の曖昧さを隠していないか。
- [ ] 略語やチーム内だけの言い回しが、public API、ログ、設定名、JSON プロパティなど長く残る境界に漏れていないか。
- [ ] コレクション名が、単数/複数、重複可否、順序、null/empty の意味を誤解させないか。

## 責務分割と変更しやすさ

- [ ] 1 つのメソッドに、入力検証、変換、永続化、通知、ログなどが詰め込まれていないか。
- [ ] 分割後のメソッド名が `CreateX`、`ValidateX`、`CalculateX` のように意図を表し、`DoX`、`HandleX` に寄りすぎていないか。
- [ ] 変更理由が局所化され、仕様変更、表示変更、永続化変更で同じメソッドを何度も直す構造になっていないか。
- [ ] private helper が増えた結果、状態変更の順序、例外時の扱い、トランザクション境界が追いにくくなっていないか。
- [ ] テストしたい判断が、現在時刻、乱数、環境変数、HTTP、DB、ファイル I/O から分離されているか。
- [ ] DTO、entity、domain model、view model の変換責務が混ざり、同じ null/format 変換が複数箇所に散っていないか。
- [ ] 新しい抽象化が、実装を隠すだけでなく、差し替え、テスト、境界の明確化に効いているか。

## 条件式と分岐

- [ ] 複雑な条件式を、意味のある変数やメソッド名で説明できているか。
- [ ] `&&` と `||` が混ざる条件に、読み間違いを防ぐ括弧、名前付き変数、または guard clause があるか。
- [ ] 早期 return により正常系と異常系の流れが読みやすくなっているか。
- [ ] null、空コレクション、境界値、権限不足、外部サービス失敗などの失敗条件が明示されているか。
- [ ] 条件追加により、既存ケースの優先順位や排他関係が崩れていないか。
- [ ] `switch` や pattern matching で、未処理ケースを `_` に押し込めず、ドメイン上の default を説明しているか。
- [ ] null チェック後に別スレッド、再取得、`await` を挟み、前提が変わる可能性を見落としていないか。

## コメントと documentation

- [ ] コメントは「何をしているか」ではなく、背景、制約、例外的な判断を説明しているか。
- [ ] 古いコメントが実装と食い違っていないか。
- [ ] public API の XML documentation が、引数、戻り値、例外、null の扱いを利用者目線で説明しているか。
- [ ] TODO や暫定対応に、期限、理由、削除条件、関連 issue などの手がかりがあるか。
- [ ] コメントで補うより、型やメソッド名で表現した方が安全な箇所はないか。
- [ ] ログメッセージや例外メッセージが、原因調査に必要な識別子を含み、機密情報や巨大な値を出していないか。

## Analyzer、EditorConfig、nullable

- [ ] analyzer の警告を、単なる雑音ではなく設計、API 契約、例外安全性のヒントとして確認しているか。
- [ ] `.editorconfig` の変更が、個人の好みではなくチーム全体の保守性として説明でき、既存コードへの影響範囲が分かるか。
- [ ] `!` null-forgiving operator は、null でないことを証明できる局面だけに限定されているか。
- [ ] `#pragma warning disable`、`SuppressMessage`、`.editorconfig` の severity 変更は、範囲が最小で理由が残っているか。
- [ ] nullable 注釈が、DB、JSON、外部 API、古いライブラリとの境界で実態に合っているか。
- [ ] `string?`、`IEnumerable<T>?`、空文字、空配列のどれで「値なし」を表すかが API ごとに一貫しているか。
- [ ] `required`、constructor、factory、serializer の組み合わせで、初期化漏れを warning だけに任せていないか。
- [ ] `MemberNotNull`、`NotNullWhen` などの属性を使う場合、実装が属性の契約を本当に満たしているか。
- [ ] CI で失敗させる warning と、IDE の suggestion に留める rule が区別されているか。
- [ ] analyzer 対応のためだけに可読性や設計を悪化させていないか。必要ならルール側を調整する判断ができるか。

## レビューコメントの品質

- [ ] 指摘が correctness、readability、maintainability、testability、performance などの観点に分けられているか。
- [ ] blocker と suggestion の温度感が明確で、修正必須か改善提案かが伝わるか。
- [ ] 「読みにくい」だけで終わらず、読み手が誤解するリスクや変更時の問題を説明しているか。
- [ ] 好みのスタイル指摘は、必要なら analyzer、formatter、`.editorconfig` へ寄せる提案になっているか。
- [ ] レビュー中に、今回の目的から外れた大きなリファクタリングを要求していないか。
- [ ] 指摘に修正案を添える場合、現行設計で自然に入る小さな変更になっているか。

## 避けたいサイン

- [ ] warning を消すためだけの `!`、空の `catch`、理由のない抑制がある。
- [ ] 名前が短すぎる、または抽象的すぎて、ドメイン上の意味を失っている。
- [ ] メソッド分割により、逆に状態変更の順序や例外時の扱いが見えにくくなっている。
- [ ] コメントが仕様の代わりになっており、テストや型で守られていない。
- [ ] レビューがフォーマット指摘に偏り、仕様、失敗時の挙動、テスト容易性を見落としている。
- [ ] nullable や analyzer の警告対応が、根本原因ではなく表面上の警告数だけを減らしている。
