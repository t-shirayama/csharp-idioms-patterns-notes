# Part 9: スタイル、レビュー、保守性

スタイルは好みを揃えるためだけのものではありません。読み手がコードの意図をすばやく判断し、変更時の影響範囲を見積もり、レビューで本質的な問題に集中するための土台です。

実務では、命名、コメント、メソッド分割、条件式、レビュー観点、analyzer のルールを別々に扱うよりも、「次に読む人が安全に変更できるか」という基準でつなげて考えると判断しやすくなります。

この Part では、可読性を単なる美しさではなく、仕様の確認、テストのしやすさ、変更時の安全性に結びつけて扱います。レビューで迷ったときは、好みか、リスクか、チームで機械化できるルールかを分けて考えます。

## チェックリストの使い方

この Part のチェックリストは、命名やスタイルの指摘を細かく増やすためではなく、レビューで見落とすと後から修正コストが高くなる観点を確認するために使います。特に、public API、nullable 境界、analyzer 警告の抑制、責務が増えたメソッドは、差分単位で理由を説明できるかを見ます。

- 命名やコメントは、読み手が仕様、単位、null の可能性、失敗時の扱いを誤解しないかで判断する。
- メソッド分割や責務分割は、テストしたい判断と外部依存が混ざっていないかを中心に見る。
- analyzer、EditorConfig、nullable は、好みの議論を機械化できるルールと、設計上の警告として扱うべきものに分ける。
