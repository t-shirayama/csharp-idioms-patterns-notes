# Part 9: スタイル、レビュー、保守性

スタイルは好みを揃えるためだけのものではありません。読み手がコードの意図をすばやく判断し、変更時の影響範囲を見積もり、レビューで本質的な問題に集中するための土台です。

実務では、命名、コメント、メソッド分割、条件式、レビュー観点、analyzer のルールを別々に扱うよりも、「次に読む人が安全に変更できるか」という基準でつなげて考えると判断しやすくなります。

この Part では、可読性を単なる美しさではなく、仕様の確認、テストのしやすさ、変更時の安全性に結びつけて扱います。レビューで迷ったときは、好みか、リスクか、チームで機械化できるルールかを分けて考えます。
