# Part 9: スタイル、レビュー、保守性

スタイルは好みを揃えるためだけのものではありません。読み手がコードの意図をすばやく判断し、変更時の影響範囲を見積もり、レビューで本質的な問題に集中するための土台です。

この Part では、命名、コメント、責務分割、条件式、コードレビュー、analyzer / EditorConfig / nullable 警告を、すべて「次に読む人が安全に変更できるか」という基準で扱います。好みで直すもの、リスクとして直すもの、機械化して人間のレビューから外すものを分けて考えます。

## このPartで身につくこと

- 命名を、実装の雰囲気ではなく責務、単位、状態、失敗時の扱いが伝わるかで判断できる。
- コメントと XML documentation を、コードの言い換えではなく境界、制約、public API の契約を補う手段として使える。
- メソッド分割、責務分割、条件式の読みやすさを、テストしやすさと変更しやすさに結びつけてレビューできる。
- analyzer、EditorConfig、nullable 警告を、チームの好み、設計上の警告、例外的な抑制に切り分けられる。

## 重要トピック

命名では、`Manager`、`Data`、`Info` のような広すぎる名前を避け、何を判断し、何を保持し、どの単位を扱うかが分かる名前を優先します。boolean は肯定形で読みやすくし、時刻、金額、件数、ID などは単位や意味が落ちない名前にします。

コメントは「なぜそうするのか」「呼び出し側が守るべき契約は何か」「例外や null をどう扱うか」を残すために使います。処理をそのまま説明するコメントは、コード変更時に嘘になりやすいため、名前や分割で解消できないかを先に考えます。

レビューでは、表面的なスタイル指摘を analyzer / EditorConfig に寄せ、人間は仕様、境界、null、例外、責務、テスト観点に集中します。nullable 警告の抑制は、警告を消すこと自体ではなく、null が来ない理由をコード上で説明できるかを基準にします。

## 実務での使いどころ

- PR レビューで、好みの指摘と不具合リスクの指摘を分けたいとき。
- public API、DTO、設定、ログ、例外メッセージなど、名前が利用者の理解に直結するとき。
- メソッドが肥大化し、テストしたい判断と外部 I/O が混ざり始めたとき。
- nullable 警告や analyzer 警告を、抑制するか設計を直すか判断したいとき。

## よくある落とし穴

- 命名を短さだけで決め、単位、状態、失敗時の意味が読み手に伝わらない。
- コメントで実装をなぞり、コード変更後に古い仕様説明だけが残る。
- メソッド分割を行数だけで判断し、責務やテスト境界が改善されていない。
- analyzer 警告を一括抑制し、実際の null リスクや API 契約の曖昧さを隠してしまう。
- レビューでフォーマット指摘が増え、仕様、例外、境界値、保守性の確認が後回しになる。

## レビュー観点

- 名前から、対象、責務、単位、null の可能性、成功/失敗の意味が読めるか。
- コメントは、コードでは表しにくい意図、制約、互換性、外部仕様を説明しているか。
- 条件式は、正常系と異常系、早期 return、複雑な boolean の分解が適切か。
- メソッドやクラスは、変更理由が複数混ざっていないか。テストしたい判断を単独で検証できるか。
- analyzer、EditorConfig、nullable の警告は、機械的に直せるもの、人間が設計判断するもの、理由付きで抑制するものに分けられているか。
