# Part 14 レビュー用チェックリスト

並行処理やバックグラウンド処理を含む変更では、正常時に速く動くかだけでなく、共有状態、排他範囲、同時実行数、停止、例外、キュー滞留を確認する。特に、再現しにくいバグほどレビュー時の設計確認が効く。

## 共有状態と thread safety

- [ ] 複数スレッドや複数タスクから触られる mutable state が明確になっている。
- [ ] `List<T>`、`Dictionary<TKey,TValue>`、カウンター、キャッシュ、乱数などを無防備に共有していない。
- [ ] 共有状態をなくせる場合は、ローカル変数、不変データ、メッセージ渡しで置き換えている。
- [ ] `lock` の対象オブジェクトが private で、外部から同じロックを取られない。
- [ ] `lock` の中で重い処理、外部 I/O、長時間の計算をしていない。
- [ ] 複数の値をまとめて更新する場合、途中状態を他の処理から見られないようにしている。

## 排他と同時実行数

- [ ] 排他したい処理と、同時実行数を制限したい処理を分けて考えている。
- [ ] `SemaphoreSlim` の `WaitAsync` と `Release` が `try` / `finally` で対応している。
- [ ] 同時実行数の値が、DB 接続数、外部 API 制限、CPU、メモリと矛盾していない。
- [ ] `Task.WhenAll` で入力件数分の処理を無制限に開始していない。
- [ ] キャンセル時に待機中の処理と実行中の処理の扱いが決まっている。

## concurrent collection

- [ ] concurrent collection を使う理由が、単なる安心感ではなく、複数 producer / consumer などの要件に合っている。
- [ ] `ContainsKey` の後に `Add` するような check-then-act を避け、`GetOrAdd`、`TryAdd`、`AddOrUpdate` を使っている。
- [ ] `GetOrAdd` の factory が複数回呼ばれる可能性を考慮している。
- [ ] 複数操作をまたぐ業務上の不変条件を、collection 単体の thread safety に任せていない。
- [ ] 列挙中のスナップショット性や順序保証に依存していない。

## BackgroundService と定期実行

- [ ] `ExecuteAsync` が `stoppingToken` を監視し、停止要求に反応する。
- [ ] `PeriodicTimer.WaitForNextTickAsync` にキャンセルを渡している。
- [ ] 前回処理が終わる前に次回処理が重ならない設計になっている。
- [ ] 例外時にサービス全体を止めるのか、ログを出して継続するのか決まっている。
- [ ] アプリ終了時に処理中ジョブを中断、完了待ち、再実行対象のどれにするか決めている。
- [ ] 定期処理のログ、メトリクス、処理時間、失敗理由を追える。

## Channel と producer / consumer

- [ ] producer と consumer の責務が分かれている。
- [ ] bounded channel を検討し、キュー長や `FullMode` が要件に合っている。
- [ ] `WriteAsync`、`ReadAllAsync`、ワーカー処理に `CancellationToken` が渡っている。
- [ ] producer 完了時に `Complete` し、consumer が終了できる。
- [ ] consumer 例外時に channel を閉じる、再試行する、失敗キューに移すなどの方針がある。
- [ ] 処理順序、重複処理、冪等性、部分失敗の扱いが明確になっている。

## テストと観測性

- [ ] 並行処理のテストが固定の `Thread.Sleep` に依存していない。
- [ ] 競合、キャンセル、タイムアウト、キュー満杯、consumer 例外をテストしている。
- [ ] 同時実行数やキュー長を小さく設定して、境界条件を再現できる。
- [ ] ログに worker 名、job ID、queue length、処理時間、失敗理由などが含まれている。
- [ ] メトリクスやログから、処理遅延、滞留、失敗率、再試行の増加を追える。

## レビューコメントの観点

- [ ] 「thread-safe ではない」だけでなく、どの状態が誰から同時に触られるかを書ける。
- [ ] `lock`、`SemaphoreSlim`、`Channel` のどれを提案するか、解きたい問題に合わせて説明できる。
- [ ] 「並列化すべき」ではなく、上限、順序、副作用、失敗時の扱いも合わせて確認している。
- [ ] バックグラウンド処理では、開始時より停止時と例外時の挙動を重点的に見る。
