# 環境変数

## 概要

環境変数は、実行環境ごとの設定をコード外から渡すための仕組みです。
接続先、機能フラグ、認証情報の参照名などに使えますが、どこからでも直接読むとテストと変更が難しくなります。

実務では「読む場所を少なくする」「必須値を起動時に検証する」「secret を値として扱わず露出させない」ことが重要です。
環境変数はプロセス全体の状態なので、単体テストや並列実行では特に扱いに注意します。

```csharp
public sealed record AppSettings(string ApiEndpoint, string EnvironmentName);

public static AppSettings LoadSettings()
{
    var endpoint = Environment.GetEnvironmentVariable("APP_API_ENDPOINT");
    var environmentName = Environment.GetEnvironmentVariable("APP_ENV") ?? "Development";

    if (string.IsNullOrWhiteSpace(endpoint))
    {
        throw new InvalidOperationException("APP_API_ENDPOINT is required.");
    }

    return new AppSettings(endpoint, environmentName);
}
```

## 判断基準

- デプロイ先ごとに変わる値は環境変数に向いている。
- 実行中に頻繁に変わる設定は、環境変数より設定サービスや reload 可能な設定を検討する。
- 値そのものが secret の場合は、環境変数に置くより secret store から参照する設計を優先する。
- ライブラリ内部で直接読むより、アプリケーション起動時に設定型へ変換して渡す。
- 文字列、bool、数値、URI などは読み取った直後に型変換と検証を済ませる。

## 使いどころ

- 開発、検証、本番で異なる接続先やモードを渡す。
- コンテナ、CI、クラウド実行環境から設定を注入する。
- テストでは読み取り口をラップし、任意の設定に差し替える。
- feature flag の初期値や診断ログの出力レベルを環境ごとに変える。

## よい例

環境変数名を集約し、読み取り後に設定型へ閉じ込めます。

```csharp
public sealed record RetrySettings(int MaxRetries, TimeSpan Delay);

public static RetrySettings LoadRetrySettings()
{
    var rawMaxRetries = Environment.GetEnvironmentVariable("APP_MAX_RETRIES") ?? "3";

    if (!int.TryParse(rawMaxRetries, out var maxRetries) || maxRetries < 0)
    {
        throw new InvalidOperationException("APP_MAX_RETRIES must be a non-negative integer.");
    }

    return new RetrySettings(maxRetries, TimeSpan.FromSeconds(2));
}
```

## 避けたい書き方

- 処理の途中で何度も `Environment.GetEnvironmentVariable` を直接呼ぶ。
- secret の値をログ、例外、画面表示に出す。
- 未設定時に空文字で処理を続け、後段で分かりにくい失敗にする。
- 大文字小文字の違いや未設定時の既定値を仕様として決めず、環境差に任せる。
- テストで環境変数を書き換えたあと、元に戻さない。

```csharp
// 避けたい例: 失敗箇所が後段にずれ、ログにも secret が出やすい。
var connectionString = Environment.GetEnvironmentVariable("DB_CONNECTION");
logger.LogInformation("Connecting to {ConnectionString}", connectionString);
using var connection = new SqlConnection(connectionString);
```

## テストしやすくする

`Environment` を直接読む処理を設定読み込みに限定すると、業務ロジックは plain object でテストできます。
環境変数をテスト内で変更する場合は、並列実行を避けるか、変更前の値を必ず復元します。

```csharp
public interface IEnvironmentReader
{
    string? Get(string name);
}

public sealed class ProcessEnvironmentReader : IEnvironmentReader
{
    public string? Get(string name) => Environment.GetEnvironmentVariable(name);
}
```

## レビュー観点

- 起動時や設定読み込み時に必須値の検証が行われているか。
- secret と通常設定が同じログ出力経路に乗っていないか。
- 環境変数名がアプリケーション内で散らばらず、定数や設定型にまとまっているか。
- テストで環境変数のグローバル状態に依存しすぎていないか。
- bool や数値のパース失敗時に、原因が分かる例外や診断になっているか。
- 既定値が安全側に倒れているか。本番だけ危険な挙動にならないか。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
