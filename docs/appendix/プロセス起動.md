# プロセス起動

## 概要

外部プロセスの起動は、OS コマンドや既存ツールを C# から呼び出すための境界です。
`ProcessStartInfo` で実行ファイル、引数、標準出力、標準エラー、終了コードを明示し、呼び出し元が成功と失敗を判断できる形にします。

プロセス起動は便利ですが、呼び出し先の存在、PATH、作業ディレクトリ、文字コード、権限、実行時間に依存します。
通常のメソッド呼び出しよりも失敗の種類が多いため、起動できない、タイムアウトする、終了コードだけ失敗する、出力が大きい、というケースを最初から扱います。

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = "dotnet",
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
};

startInfo.ArgumentList.Add("--info");

using var process = Process.Start(startInfo)
    ?? throw new InvalidOperationException("Failed to start process.");

var outputTask = process.StandardOutput.ReadToEndAsync();
var errorTask = process.StandardError.ReadToEndAsync();

await process.WaitForExitAsync(cancellationToken);

var output = await outputTask;
var error = await errorTask;

if (process.ExitCode != 0)
{
    throw new InvalidOperationException($"Command failed: {process.ExitCode}, {error}");
}
```

## 判断基準

- ライブラリ API で同じことができるなら、まず API 呼び出しを検討する。
- 既存 CLI の仕様が安定していて、プロセス境界で分離したい場合は起動する価値がある。
- shell の構文、リダイレクト、パイプが本当に必要な場合だけ `cmd.exe` や `sh` を経由する。
- ユーザー入力を含む場合は、文字列の command line ではなく `ArgumentList` で引数を分ける。
- 長時間実行や外部 I/O を伴う場合は、タイムアウト、キャンセル、ログ量の上限を仕様に含める。

## 使いどころ

- CLI ツール、変換ツール、既存バッチなどをアプリケーションから呼び出す。
- 標準出力を読み取り、結果を次の処理に渡す。
- 終了コードと標準エラーをログや例外に含め、運用時に原因を追いやすくする。
- 別プロセスにすることで、クラッシュや依存ライブラリの競合を呼び出し元から切り離す。

## よい例

`ArgumentList` で引数を分け、作業ディレクトリと timeout を呼び出し側から渡せるようにします。

```csharp
static async Task<string> RunDotnetInfoAsync(
    string workingDirectory,
    CancellationToken cancellationToken)
{
    using var timeout = CancellationTokenSource.CreateLinkedTokenSource(cancellationToken);
    timeout.CancelAfter(TimeSpan.FromSeconds(30));

    var startInfo = new ProcessStartInfo("dotnet")
    {
        WorkingDirectory = workingDirectory,
        RedirectStandardOutput = true,
        RedirectStandardError = true,
        UseShellExecute = false,
    };

    startInfo.ArgumentList.Add("--info");

    using var process = Process.Start(startInfo)
        ?? throw new InvalidOperationException("dotnet could not be started.");

    var outputTask = process.StandardOutput.ReadToEndAsync(timeout.Token);
    var errorTask = process.StandardError.ReadToEndAsync(timeout.Token);

    await process.WaitForExitAsync(timeout.Token);

    if (process.ExitCode != 0)
    {
        var error = await errorTask;
        throw new InvalidOperationException($"dotnet failed: {process.ExitCode}, {error}");
    }

    return await outputTask;
}
```

## 避けたい書き方

- ユーザー入力を文字列連結してコマンドラインを組み立てる。
- `UseShellExecute = true` のまま、リダイレクトや引数の扱いを曖昧にする。
- 終了コードを確認せず、プロセスが起動できたことだけで成功扱いにする。
- 標準出力だけを読み、標準エラーが詰まってプロセスが止まる可能性を残す。
- カレントディレクトリや PATH に依存し、CI やサービス実行でだけ壊れる。

```csharp
// 避けたい例: 入力値に空白や制御文字が入ると壊れやすく、注入にも弱い。
var process = Process.Start("cmd.exe", "/c tool.exe " + userInput);
```

```csharp
// 避けたい例: 起動できたことと処理成功を混同している。
using var process = Process.Start("tool.exe", "convert input.txt");
await process!.WaitForExitAsync();
return "converted";
```

## テストしやすくする

外部プロセスを直接呼ぶコードは、薄い adapter に閉じます。
業務ロジックは `ICommandRunner` のような境界に依存させると、終了コード、標準出力、標準エラー、タイムアウトを単体テストで再現できます。

```csharp
public sealed record CommandResult(int ExitCode, string Output, string Error);

public interface ICommandRunner
{
    Task<CommandResult> RunAsync(string fileName, IReadOnlyList<string> args, CancellationToken ct);
}
```

## レビュー観点

- `ArgumentList` などで引数が分離され、コマンドインジェクションに弱い文字列連結になっていないか。
- 標準出力と標準エラーを読み取り、デッドロックしない順序になっているか。
- `ExitCode`、タイムアウト、キャンセル時の扱いが呼び出し側に伝わるか。
- 実行ファイルのパスや作業ディレクトリが環境差に依存しすぎていないか。
- ログに出す出力が大きすぎないか、secret や個人情報を含まないか。
- Windows / Linux のパス、改行、実行ファイル名の違いを考慮しているか。
