# バージョン情報

## 概要

.NET には assembly version、file version、informational version など複数のバージョンがあります。
互換性判断、ファイル表示、画面やログへの表示では目的が違うため、どの値を使うかを分けて考えます。

実務では、障害調査で「どの commit が動いていたか」を特定できることが重要です。
表示用の SemVer、アセンブリ互換性用の `AssemblyVersion`、ファイル情報用の `FileVersion`、診断用の commit hash を混同しないようにします。

```csharp
var assembly = typeof(Program).Assembly;
var version = assembly
    .GetCustomAttribute<AssemblyInformationalVersionAttribute>()
    ?.InformationalVersion
    ?? assembly.GetName().Version?.ToString()
    ?? "unknown";

logger.LogInformation("Application version: {Version}", version);
```

## 判断基準

- ユーザーや運用者に見せる値は `AssemblyInformationalVersion` を優先する。
- ライブラリの binary compatibility に影響する値は `AssemblyVersion` として慎重に管理する。
- Windows のファイルプロパティやインストーラー向けには `AssemblyFileVersion` を使い分ける。
- Web API やバッチでは、起動ログと診断 endpoint の両方で同じ値を確認できるようにする。
- 公開版では内部ブランチ名やビルドパスを出さず、必要なら commit hash の短縮値にとどめる。

## 使いどころ

- 障害調査用に起動ログへアプリケーションバージョンを出す。
- 画面や API の診断エンドポイントで、デプロイ済みの版を確認できるようにする。
- CI で Git commit hash やタグを `InformationalVersion` に埋め込む。
- 複数サービスの問い合わせ時に、互いの期待バージョンをログで照合する。

## よい例

取得処理を小さくまとめ、表示用バージョンがない場合だけ fallback します。

```csharp
static string GetDisplayVersion(Assembly assembly)
{
    var informational = assembly
        .GetCustomAttribute<AssemblyInformationalVersionAttribute>()
        ?.InformationalVersion;

    if (!string.IsNullOrWhiteSpace(informational))
    {
        return informational;
    }

    return assembly.GetName().Version?.ToString() ?? "unknown";
}
```

`.csproj` では `Version` と `InformationalVersion` を分け、CI から commit hash を渡す運用にすると、手作業の更新漏れを減らせます。

## 避けたい書き方

- 表示用バージョンに `AssemblyVersion` だけを使い、ビルドやコミットの識別ができない。
- 手作業で複数ファイルのバージョン文字列を更新する。
- バージョン情報にブランチ名や内部情報を無条件で公開する。
- 起動ログには出しているが、調査時に API や管理画面から確認できない。
- version をコード内の定数として持ち、リリース手順とずれていく。

```csharp
// 避けたい例: リリース作業で更新漏れが起きやすい。
public const string ApplicationVersion = "1.4.0";
```

## テストしやすくする

バージョン取得自体は薄い処理なので、業務ロジックに直接散らさず `IVersionProvider` のような境界へ寄せます。
診断 endpoint のテストでは値そのものより、空でないこと、期待するフィールド名で返ること、secret を含まないことを確認します。

```csharp
public interface IVersionProvider
{
    string DisplayVersion { get; }
}
```

## レビュー観点

- 表示用、互換性用、ファイル情報用のバージョンを混同していないか。
- リリース手順や CI で自動的に値が埋まるか。
- ログや診断 API で、問い合わせ時に実行中の版を特定できるか。
- 公開してよい情報と内部向け情報の境界が決まっているか。
- 複数 assembly を含む場合に、どの assembly のバージョンを代表として扱うか決まっているか。
- container image tag、package version、application version が大きく食い違わない運用になっているか。
