# その他の実務定石

この補足編では、日々の業務アプリケーションで主役になりにくいものの、境界処理や運用時の不具合に直結しやすい定石を扱います。
プロセス起動、環境変数、バージョン情報、ZIP、reflection、source generator、unsafe / native interop は、どれも「少し書ける」だけでは足りず、失敗時の見え方、テストしやすさ、保守範囲を意識して使う必要があります。

## このAppendixで身につくこと

- OS、ファイル、実行環境など、アプリケーション外側の境界を安全に扱う判断基準
- 環境差分、権限、文字コード、パス、タイムアウトなど、運用で表面化しやすい前提の見つけ方
- reflection、source generator、unsafe / native interop のような強い機能を、使う理由と閉じ込め方から判断する視点
- 本体コード、テスト、運用メモのどこに前提を残すべきかをレビューする観点

## 読む順番

1. [環境変数](環境変数.md) で、実行環境から値を受け取るときの既定値、検証、秘匿情報の扱いを確認する。
2. [プロセス起動](プロセス起動.md) で、外部コマンドの引数、標準出力、終了コード、タイムアウトをどう扱うかを見る。
3. [ZIP 操作](zip-操作.md) と [バージョン情報](バージョン情報.md) で、配布物や実行時情報を扱うときの境界を押さえる。
4. [reflection](reflection.md) と [source generator の入口](source-generator-の入口.md) で、メタプログラミングを導入する判断基準を整理する。
5. [unsafe / native interop の入口](unsafe-native-interop-の入口.md) で、通常のC#から外れる領域を最小範囲に閉じ込める考え方を確認する。
6. 最後に [補足トピックのレビュー用チェックリスト](checklist.md) で、入力、失敗時の見え方、テスト境界、運用差分を点検する。

## 重要トピック

- 外部入力は「信頼できる設定」ではなく、検証対象として扱う。
- 文字列で表現されるパス、コマンド、環境変数、型名は、早めに意味のある型や設定オブジェクトへ変換する。
- reflection や unsafe は便利さだけで採用せず、呼び出し範囲、代替案、テスト方法、失敗時のログを先に決める。
- source generator は実行時の複雑さをコンパイル時へ移す道具なので、生成物のデバッグしやすさも設計に含める。

## 実務での使いどころ

- バッチ、CLI、CI/CD、運用スクリプトから外部プロセスを呼び出す。
- Docker、クラウド、オンプレミスなど、環境ごとに設定値が変わるアプリケーションを作る。
- 配布物の展開、プラグイン探索、バージョン表示、診断ログなど、アプリ本体の外周を整える。
- 高速化、相互運用、コード生成が必要になったときに、影響範囲を限定して強い機能を使う。

## よくある落とし穴

- コマンド引数を文字列連結で組み立て、空白や引用符、特殊文字で壊れる。
- 環境変数が存在する前提で読み、空文字、未設定、型変換失敗を区別できない。
- ZIP 展開時に意図しないパスへ書き込む可能性を見落とす。
- reflection の失敗を実行時まで隠し、リネームや trimming、AOT で壊れる。
- unsafe や native interop の境界が広がり、通常のC#コードまでメモリ安全性の前提を失う。

## レビュー観点

- 外部入力の検証、既定値、エラー表示、ログが揃っているか。
- OS差分、パス区切り、文字コード、権限、カレントディレクトリに依存していないか。
- タイムアウト、キャンセル、リトライ、後片付けの方針が明確か。
- 強い機能を使う理由がコード上またはドキュメント上で説明できるか。
- テストで差し替える境界があり、実環境なしでも主要な失敗ケースを確認できるか。
