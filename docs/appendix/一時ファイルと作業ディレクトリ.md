# 一時ファイルと作業ディレクトリ

一時ファイルは、失敗時の後片付け、権限、同時実行、atomic replace を考えないと、本番環境でディスク枯渇やデータ破損の原因になります。

## 基本方針

- `Path.GetTempPath()` 直下に固定名を置かず、処理単位の一意な作業ディレクトリを作る。
- 作業ディレクトリは `try` / `finally` で削除する。ただし障害調査用に保存する設定を持たせる場合は、保存条件と期限を決める。
- 完成前のファイルを公開場所に直接書かず、同一ボリューム上で一時ファイルへ書いてから `File.Move` / `File.Replace` で切り替える。
- ユーザー入力をファイル名に使う場合は、予約名、区切り文字、長さ、大小文字衝突を正規化する。

```csharp
var workDir = Path.Combine(Path.GetTempPath(), "import-" + Guid.NewGuid().ToString("N"));
Directory.CreateDirectory(workDir);

try
{
    var tempFile = Path.Combine(workDir, "result.tmp");
    await File.WriteAllTextAsync(tempFile, content, cancellationToken);

    File.Move(tempFile, destinationPath, overwrite: true);
}
finally
{
    Directory.Delete(workDir, recursive: true);
}
```

## 並列実行時の衝突

避けたいのは、固定の `work` ディレクトリや `output.tmp` を複数ジョブで共有する書き方です。並列実行、再試行、前回失敗の残骸が重なると、別リクエストのデータを上書きします。

```csharp
// 避けたい例: 複数実行で衝突する。
var tempFile = Path.Combine(Path.GetTempPath(), "output.tmp");
```

ジョブ ID、テナント ID、日時だけでは衝突や推測可能性を避けきれないため、乱数または GUID を含めます。

## 権限と公開範囲

一時ディレクトリには機密データが残りやすいため、ログにフルパスや内容を出しすぎないようにします。コンテナでは `/tmp` の容量制限、read-only filesystem、emptyDir の size limit も運用前提として確認します。

## レビュー観点

- 固定名の一時ファイルで並列実行時に衝突しないか。
- 失敗、キャンセル、例外時にも cleanup されるか。
- 完成前のファイルを利用者が読めないよう atomic replace しているか。
- 一時領域の容量、権限、機密データの残存を運用で監視できるか。
