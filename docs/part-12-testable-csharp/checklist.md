# Part 12: テストしやすい C# のレビュー checklist

このチェックリストは、C# の設計とテストコードをレビューするときに、仕様の見えやすさ、失敗時の再現性、保守しやすさを確認するためのものです。テスト数を増やすことよりも、何を保証しているかを説明できる状態を目指します。

レビューでは、対象コードの変更理由に近い項目から確認します。小さな単体テストで見るべきこと、結合テストで見るべきこと、手動確認でしか見ていないことを分けて考えると、抜け漏れを見つけやすくなります。

## テスト対象の切り分け

- [ ] 業務判断、変換、計算などの純粋な処理が、小さく直接テストできる形になっている。
- [ ] DB、HTTP、ファイル、時刻、乱数などの副作用が、境界に寄せられている。
- [ ] private method を無理に直接テストしたくなる設計になっていない。公開される振る舞いから確認できる。
- [ ] controller や framework 依存の薄い層ではなく、仕様判断を持つ層にテストが置かれている。
- [ ] 設定値、feature flag、権限、tenant などの文脈が、暗黙の static 参照ではなくテストで差し替えられる。
- [ ] テストのためだけに本番コードの責務が不自然に漏れていない。

## テストケース設計

- [ ] 正常系だけでなく、null、空、境界値、重複、権限不足、外部失敗などの代表的な失敗系を含んでいる。
- [ ] テスト名から、条件、操作、期待結果が読み取れる。
- [ ] 1つのテストが複数の仕様をまとめて検証しすぎていない。
- [ ] 実装詳細ではなく、利用者や呼び出し元から見た振る舞いを検証している。
- [ ] 同値クラス、境界値、状態遷移、権限差分など、仕様の種類に応じてケースを選んでいる。
- [ ] API や DB の変更では、成功時だけでなく 400、401、403、404、409、500 相当の扱いを必要に応じて確認している。
- [ ] バグ修正時は、再発を防ぐ最小のテストが追加されている。

## Arrange とテストデータ

- [ ] arrange が長すぎず、テストで重要な値と既定値が区別できる。
- [ ] テストデータビルダーや factory の既定値が、偶然失敗しない妥当な値になっている。
- [ ] object mother 的な共有データが肥大化し、テスト間の意図を見えにくくしていない。
- [ ] ランダム値や現在時刻に依存せず、毎回同じ条件で再現できる。
- [ ] ID、日付、金額、タイムゾーン、カルチャなど、失敗しやすい値はテスト上で明示されている。
- [ ] assertion は「何が壊れたか」が分かる粒度になっており、複数要素の比較では順序や差分が読み取れる。

## mock / stub / fake

- [ ] mock は相互作用を検証したい境界に限定し、内部実装の呼び出し順を過剰に固定していない。
- [ ] stub は入力条件を作るために使い、検証対象の本質を隠していない。
- [ ] fake は本物に近い振る舞いを持つが、本番の DB や外部サービスと同じ保証があると誤解していない。
- [ ] 所有していない外部 API を直接 mock せず、自分たちが管理する adapter や port を境界にしている。
- [ ] mock の設定が複雑すぎる場合、設計上の依存が太くなっていないか見直している。
- [ ] `Verify` の回数や呼び出し順の検証が、仕様ではなく実装都合を固定していない。
- [ ] EF Core InMemory、SQLite、Testcontainers などの選択が、確認したい制約、transaction、SQL 差分に合っている。

## 時刻・乱数・外部 I/O

- [ ] `DateTime.Now`、`Guid.NewGuid()`、乱数生成、環境変数参照が業務ロジックの中に直接散らばっていない。
- [ ] 時刻は `TimeProvider` や clock 抽象などで注入でき、タイムゾーンと基準時刻の扱いが明確になっている。
- [ ] ID 生成や乱数は、テストで固定できる境界を持っている。
- [ ] HTTP、DB、ファイル I/O の失敗、タイムアウト、キャンセルをテストで表現できる。
- [ ] 外部 API の再試行、タイムアウト、レート制限、部分失敗を fake や handler で再現できる。
- [ ] integration test と unit test の責務が混ざり、遅い外部依存で日常的な確認が重くなっていない。

## 例外・非同期・キャンセル

- [ ] 例外テストは型だけでなく、必要に応じてメッセージ、エラーコード、内包情報を確認している。
- [ ] 非同期処理は `await` され、fire-and-forget による見逃しがない。
- [ ] `CancellationToken` を受け取る API では、下位の非同期呼び出しへ渡している。
- [ ] タイムアウト、キャンセル、リトライの責務がテストで確認できる場所にある。
- [ ] 並行処理のテストは、sleep に頼りすぎず、同期ポイントや fake を使って再現性を確保している。
- [ ] `Assert.ThrowsAsync` や失敗タスクの検証で、同期例外と非同期例外を取り違えていない。
- [ ] キャンセル時に状態更新、ログ、外部送信が途中で残らないことを、必要な境界で確認している。

## API・DB・設定のテスト

- [ ] API 契約テストでは、status code、response body、validation error、header など、クライアントが依存する要素を確認している。
- [ ] DB を伴うテストでは、一意制約、外部キー、transaction、同時更新など、単純な repository mock では見えない失敗を押さえている。
- [ ] 設定や Options のテストでは、必須設定漏れ、不正な範囲、環境ごとの差分、secret の扱いを確認している。
- [ ] ログを検証する場合は、文言全文ではなく、レベル、イベント、構造化プロパティ、例外有無など保守しやすい単位で見ている。
- [ ] migration や seed の変更は、空 DB だけでなく既存データがある状態の確認方法を持っている。

## テストコードの保守性

- [ ] テストは本番コードの読み替えではなく、仕様の例として読める。
- [ ] 共通化により、各テストの条件や期待結果が見えなくなっていない。
- [ ] fixture の共有状態がテスト順序依存や並列実行時の不安定さを生んでいない。
- [ ] テスト失敗時に、どの仕様が壊れたかを短時間で追える。
- [ ] カバレッジの数値だけでなく、重要な分岐と境界を押さえている。
- [ ] 低速なテスト、外部依存が必要なテスト、通常の unit test が分類され、CI での実行範囲が分かる。

## レビューで止めて確認したい兆候

- [ ] mock の setup が本体ロジックより読みにくい。
- [ ] テストを追加するために、本番コードへ不自然な setter や分岐が増えている。
- [ ] テストが頻繁に flaky になり、原因が時刻、並行実行、外部 I/O、共有状態にありそう。
- [ ] 重要な仕様変更なのに、正常系のスナップショットや画面確認だけで済ませている。

## TODO対応: flaky test の原因別チェック

> 対応元: P1 / flaky test の原因別チェック、並列実行、共有 DB、時計、ランダム、外部 API、順序依存の切り分け項目を独立節として追加する。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
