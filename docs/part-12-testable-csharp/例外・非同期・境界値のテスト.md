# 例外・非同期・境界値のテスト

## 概要

例外、非同期、境界値は、実装者の想定漏れが表に出やすい領域です。正常系だけでなく、null、empty、invalid input、キャンセル、タイムアウト、外部依存の失敗を、どの層で扱うか決めてテストします。例外は「投げること」だけでなく、例外型、パラメーター名、呼び出し元が回復できる情報も確認します。

## 使いどころ

- 入力検証で `ArgumentNullException` や `ArgumentOutOfRangeException` を使う場面。
- `async` method が成功、失敗、キャンセルを正しく `Task` に反映するか確認したい場面。
- 最小値、最大値、空配列、重複、端数など、仕様の境界が障害につながりやすい場面。

```csharp
[Fact]
public async Task LoadAsync_throws_when_user_does_not_exist()
{
    var service = new UserProfileService(new MissingUserRepository());

    var exception = await Assert.ThrowsAsync<UserNotFoundException>(
        () => service.LoadAsync(Guid.NewGuid(), CancellationToken.None));

    Assert.Contains("User", exception.Message);
}
```

## 例外テスト

例外テストでは、例外型だけでなく、必要に応じて `ParamName`、エラーコード、対象 ID なども確認します。メッセージ全文は文言変更に弱いため、仕様として重要な場合だけ確認します。

```csharp
[Fact]
public void Constructor_throws_when_quantity_is_negative()
{
    var exception = Assert.Throws<ArgumentOutOfRangeException>(
        () => new OrderLine(productId: Guid.NewGuid(), quantity: -1));

    Assert.Equal("quantity", exception.ParamName);
}
```

業務例外では、呼び出し元が回復や表示に使う情報を確認します。

```csharp
var exception = await Assert.ThrowsAsync<OrderNotFoundException>(
    () => service.LoadAsync(orderId, CancellationToken.None));

Assert.Equal(orderId, exception.OrderId);
```

## 非同期テスト

非同期メソッドは `await` して検証します。`.Result` や `.Wait()` は例外を包んだり、同期コンテキストでデッドロックを招いたりするため避けます。

```csharp
[Fact]
public async Task SaveAsync_passes_cancellation_token_to_repository()
{
    using var cts = new CancellationTokenSource();
    var repository = new SpyOrderRepository();
    var service = new OrderService(repository);

    await service.SaveAsync(new OrderBuilder().Build(), cts.Token);

    Assert.Equal(cts.Token, repository.ReceivedCancellationToken);
}
```

キャンセルは業務エラーではありません。`OperationCanceledException` を握りつぶして成功扱いにしたり、`Error` ログに積んだりしない方が、呼び出し元やホストのキャンセル制御と合います。

## 境界値の選び方

境界値は、仕様が変わる点の前後を選びます。数値なら最小、最大、ちょうど境界、境界の 1 つ外。コレクションなら空、1 件、複数、重複。文字列なら null、空、空白、最大長、最大長超過を確認します。

```csharp
[Theory]
[InlineData(0, false)]
[InlineData(1, true)]
[InlineData(100, true)]
[InlineData(101, false)]
public void IsValidQuantity_returns_expected_result(int quantity, bool expected)
{
    var actual = QuantityPolicy.IsValid(quantity);

    Assert.Equal(expected, actual);
}
```

パラメータ化テストは便利ですが、ケースが多すぎると意図が薄まります。仕様上の代表点に絞り、必要ならテストを分けます。

## 避けたい書き方

- 例外型を確認せず、単に「何か例外が出る」ことだけを見る。
- 非同期テストで `.Result` や `.Wait()` を使い、本来の例外やキャンセルを見失う。
- 境界値をコメントだけで説明し、テストデータやテスト名から意図が読めない。
- 例外メッセージ全文に依存し、文言修正で不要にテストが壊れる。
- cancellation を timeout や業務エラーと同じ扱いで検証する。

```csharp
// 避けたい例: AggregateException に包まれたり、同期ブロックで別の問題を起こしたりする。
Assert.Throws<UserNotFoundException>(() =>
    service.LoadAsync(userId, CancellationToken.None).Wait());
```

```csharp
// 避けたい例: 何でもよい例外を許してしまう。
Assert.Throws<Exception>(() => valueObject.Validate());
```

## レビュー観点

- 正常系、異常系、境界値のどれを検証しているテストか明確か。
- `Assert.ThrowsAsync` など、非同期に合った assertion を使っているか。
- cancellation を業務エラーとして扱っていないか。
- null、empty、invalid input の扱いが API 契約として一貫しているか。
- 例外の型、パラメーター名、エラーコードなど、呼び出し元が使う情報を確認しているか。
- 境界値が「なぜその値か」テスト名やデータから読み取れるか。
- timeout、retry、キャンセル、外部依存失敗を別々のケースとして扱っているか。

## TODO対応: Task.WhenAll

> 対応元: P0 / `Task.WhenAll`、fire-and-forget、background service、`IAsyncEnumerable<T>`、`ValueTask`、並行処理の失敗・キャンセル・部分成功のテスト観点を追加する。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
