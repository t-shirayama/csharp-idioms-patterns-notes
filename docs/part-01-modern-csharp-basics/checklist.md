# Part 1 コードレビューチェックリスト

現代 C# の土台に関するレビューでは、「その機能を使っているか」よりも、プロジェクトの実行環境、型の契約、初期化後の不変性、リソース所有権が読み取れるかを確認します。
各項目は、コード、設定、テスト、レビューコメントのいずれかで根拠を確認できる粒度にしています。

## プロジェクト前提

- [ ] `.csproj` の `TargetFramework` は、運用環境、CI、コンテナ、主要ライブラリの対応範囲と一致しているか。
- [ ] 使用している .NET API は対象 TFM で利用できるか。C# 構文だけを見て判断していないか。
- [ ] SDK バージョンは `global.json`、CI、開発環境で大きくずれていないか。
- [ ] `Nullable` と `ImplicitUsings` の有効/無効が、プロジェクト内で一貫した前提になっているか。
- [ ] Console、Web API、Worker Service などの種別に合ったログ、DI、設定読み込みの仕組みに乗っているか。
- [ ] ライブラリの TFM は、利用側に不要なアップグレードを強制しない範囲に収まっているか。

## null と型の契約

- [ ] public API の引数、戻り値、プロパティで `T` と `T?` が契約どおりに使い分けられているか。
- [ ] 外部入力、DB、JSON、古い API との境界で null を検証し、内部モデルへ入る前に非 null へ寄せているか。
- [ ] `!` null 許容抑制演算子は、直前のガード、初期化順序、フレームワークの保証など根拠がある箇所だけに使っているか。
- [ ] `default` で作れる値型が、業務上の不正状態として静かに処理されないか。
- [ ] `Try` パターン、`MemberNotNull`、`NotNullWhen` などの nullable 属性は、実装の分岐と一致しているか。
- [ ] null、空文字、未設定、検索結果なしを、仕様上区別すべき場面で同じ値に潰していないか。
- [ ] `T?` が複数の業務状態を背負いすぎている場合、Result 型や専用型に分ける必要はないか。

## 型選択

- [ ] ID とライフサイクルで識別する型は `class`、値の組み合わせで識別する型は `record` / Value Object など、同一性の基準に合う形になっているか。
- [ ] `record` の値ベース等価性は、業務上の同一性やキャッシュキーとして使っても破綻しないか。
- [ ] `struct` / `record struct` は小さく、不変に近く、コピーされても意味が変わらない型に限定されているか。
- [ ] `readonly record struct` でも `default` で不正状態が作れることを、生成時や利用時の validation で扱えているか。
- [ ] 可変 `struct` のプロパティ更新やコレクション要素更新が、コピーへの更新になっていないか。
- [ ] DTO、Entity、Value Object の責務が分かれ、外部入力の都合がドメイン型の不変条件を弱めていないか。
- [ ] `with` や deconstruction で、validation や状態遷移メソッドを通るべき変更を迂回していないか。

## 初期化と不変性

- [ ] public setter は変更が必要な状態だけに限定し、初期化後に変わらない値は `init`、get-only、コンストラクタで表しているか。
- [ ] 必須値は `required`、コンストラクタ、validation のどこで保証するかがコード上で分かるか。
- [ ] `required` で初期化漏れを防ぐだけでなく、空文字、範囲外、不正な組み合わせを別途検証しているか。
- [ ] `required` が付いた property が、初期化後も public setter で壊せる状態になっていないか。
- [ ] `init` setter に validation や正規化を書く場合、例外条件と保存される値が明確か。
- [ ] オブジェクト初期化子で、コンストラクタやファクトリが守るべき不変条件をすり抜けていないか。
- [ ] `readonly` field や get-only property で、参照先オブジェクトまで不変だと誤解させる公開方法になっていないか。
- [ ] 公開コレクションから内部状態を変更できないよう、コピー、読み取り専用ビュー、専用メソッドのいずれかで境界を置いているか。
- [ ] private setter を使う場合、状態遷移メソッドが不変条件をまとめて守っているか。

## namespace と using

- [ ] namespace はフォルダ構成、層、責務と対応し、型の置き場所を推測できるか。
- [ ] Domain、Application、Infrastructure などの依存方向に反する namespace 参照が入っていないか。
- [ ] `global using` に、業務固有の名前空間や Infrastructure 依存を広げすぎていないか。
- [ ] 不要な `using` を残して、実際には使っていない依存があるように見せていないか。
- [ ] 1 ファイル 1 namespace の単純なファイルでは、file-scoped namespace でネストを減らせるか。
- [ ] using alias が増えている場合、型名や namespace の設計を見直すべき衝突が起きていないか。

## 例外とリソース管理

- [ ] 予測できる業務エラーは validation / Result / 戻り値で扱い、処理を続けられない失敗だけを例外にしているか。
- [ ] `catch` した例外は、回復、型変換、ログ付与、再スローのいずれかの目的を持っているか。
- [ ] 例外を投げ直す場合、`throw ex;` ではなく `throw;` でスタックトレースを保っているか。
- [ ] 例外メッセージとログには調査に必要な文脈があり、secret や個人情報を含めていないか。
- [ ] `IDisposable` / `IAsyncDisposable` な型は、所有しているスコープで `using` / `await using` しているか。
- [ ] using declaration (`using var`) の寿命が、必要以上に長くなっていないか。
- [ ] 非同期 flush やトランザクション終了など、非同期解放が必要な型を同期 `using` で扱っていないか。
- [ ] 呼び出し元から受け取った `Stream` などを、メソッド内で勝手に dispose していないか。
- [ ] DI コンテナやファクトリが所有する依存を、自分で生成したリソースと同じ扱いで dispose していないか。

## テスト観点

- [ ] CI、コンテナ、ローカルで SDK / TFM 前提が揃い、ビルドとテストが同じ条件で動くか。
- [ ] nullable 境界、入力 validation、初期化漏れ、不正な値型 `default` を、それぞれ失敗ケースとして確認しているか。
- [ ] Entity と Value Object の等価性、状態遷移、不変条件を業務ルールとしてテストしているか。
- [ ] `required` や `init` で守れない実行時 validation の失敗ケースをテストしているか。
- [ ] 例外、Result、validation error の扱いが、呼び出し側の分岐としてテストしやすい形になっているか。
- [ ] リソース解放が重要な処理では、正常系だけでなく異常系でも dispose されることを確認しているか。
- [ ] 呼び出し元が所有する `Stream` などを渡した後、メソッド終了後も契約どおり使えることを確認しているか。
