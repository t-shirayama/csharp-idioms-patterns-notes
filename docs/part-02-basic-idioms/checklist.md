# Part 2 コードレビューチェックリスト

基本文法のレビューでは、短く書けているかだけでなく、失敗条件、業務ルールの名前、境界条件、将来の変更に気付ける形になっているかを確認します。
各項目は、レビューでそのまま確認できる具体的な問いとして使える粒度にしています。

## 初期化とガード節

- [ ] メイン処理に入る前に、null、空、範囲外、不正な組み合わせをガードしているか。
- [ ] 引数不正は `ArgumentException` 系、状態不正は `InvalidOperationException` など、失敗の種類に合う例外型を選んでいるか。
- [ ] 例外メッセージには、呼び出し側が直せる入力名や調査に必要な文脈が含まれているか。
- [ ] 不完全なオブジェクトを段階的に組み立てる場合、その途中状態が外へ漏れないスコープに閉じているか。
- [ ] オブジェクト初期化子で、複数プロパティ間の不変条件や validation をすり抜けていないか。
- [ ] collection expression を使う場合、プロジェクトの C# バージョンとレビュー対象の読みやすさに合っているか。
- [ ] 入力コレクションを保持する場合、必要な箇所で防御的コピーを行い、不要なコピーを増やしていないか。
- [ ] ガード節が長くなった箇所は、入力モデル、validation 層、Value Object に責務を分けられるか。
- [ ] 例外、早期 return、既定値補完の使い分けは、呼び出し側の契約と一致しているか。
- [ ] 再代入される変数は、値の変化に業務上の意味があり、スコープが十分に狭いか。

## 条件分岐

- [ ] 複雑な条件式には、`IsHighValueOrder` や `RequiresApproval` のような業務語彙の名前を付けているか。
- [ ] 条件の順序に意味がある場合、より限定的な条件を先に置き、後続条件が到達不能になっていないか。
- [ ] `if`、`switch statement`、`switch expression` の使い分けは、値の変換、副作用の有無、読みやすさに合っているか。
- [ ] `switch expression` の `_` は安全な既定値か。想定外値なら例外や明示的なエラーにしているか。
- [ ] pattern matching は、単純な `if` より条件の意味を読み取りやすくしているか。
- [ ] null チェック、型チェック、業務条件をひとつの式に詰め込みすぎず、必要に応じて分割しているか。
- [ ] 副作用を伴う処理を、短さのために `switch expression` や三項演算子へ押し込んでいないか。
- [ ] 境界値や条件の優先順位を、テストケース名から追える形で確認しているか。
- [ ] property pattern や list pattern を使うことで、単純な条件がかえって読みにくくなっていないか。
- [ ] 新しい状態や種別が増えたとき、`_` や default によって未対応を隠さない形になっているか。

## null 処理

- [ ] null が「異常」「未設定」「検索結果なし」「任意入力なし」のどれを表すのか、API 契約や変数名から読めるか。
- [ ] public API の引数と戻り値で、nullable annotation が実際の null 許容/非許容と一致しているか。
- [ ] 必須データの欠落を `?.` や `??` で静かに既定値へ変換していないか。
- [ ] `??` で入れる既定値は、業務上自然な代替値であり、障害やデータ欠落を隠していないか。
- [ ] コレクションを返す API は、空コレクションを返す方針か null を返す契約かが一貫しているか。
- [ ] 空文字、空白、null を同じ扱いにする場合、その入力仕様がガードや normalization に表れているか。
- [ ] `??=` による遅延初期化は、共有状態やマルチスレッドで競合しても問題ないか。
- [ ] `!` で警告だけを消さず、非 null になる前提をガード、変換、フレームワーク保証の近くに示しているか。
- [ ] null に複数の意味が混ざる箇所は、専用型、Result 型、状態オブジェクトへ分ける候補として扱っているか。
- [ ] 外部入力や DB 値を内部モデルへ変換する境界で、null、空文字、空白の意味を正規化しているか。

## enum と flags enum

- [ ] `None = 0` や `Unknown = 0` など、`default` 値の意味を明示しているか。
- [ ] enum はコード上で閉じた選択肢として妥当で、表示名、並び順、外部コード、振る舞いを抱え込みすぎていないか。
- [ ] enum を永続化する場合、数値保存か文字列保存かを選び、リネームや値追加時の影響を確認しているか。
- [ ] `switch` の `default` や `_` で、新しい enum 値を意図せず通常扱いに落としていないか。
- [ ] 外部 API や DB から enum に変換する境界で、不明値、廃止値、未対応値の扱いを決めているか。
- [ ] `[Flags]` enum の単独値は 2 のべき乗で、組み合わせ値は単独値の OR として定義されているか。
- [ ] flags enum の判定では、`HasFlag` やビット演算の条件がゼロ値や組み合わせ値で破綻しないか。
- [ ] 値ごとの振る舞いが増えている場合、record、class、Strategy、設定テーブルへ移す候補として扱っているか。
- [ ] 排他的な状態や状態遷移を、flags enum の組み合わせで無理に表していないか。
- [ ] enum 名を画面表示や外部契約へ直接流用していないか。

## range、index、tuple、deconstruction、匿名型

- [ ] `^1` や `..` を使う箇所は、空、1件、少数件の入力で例外や意図しない範囲にならないか。
- [ ] range / index の短さより、意図を表す変数名やメソッド名を置いた方が読みやすい箇所はないか。
- [ ] tuple は private メソッドやローカルな一時値に留まり、public API の長期的な契約になっていないか。
- [ ] tuple の要素名だけで仕様上の概念を表していないか。共有される概念なら record や専用クラスに名前を付けているか。
- [ ] deconstruction により、値の順序依存や未使用値が読み手に隠れていないか。
- [ ] 匿名型は LINQ やクエリ内部の一時表現に留まり、メソッド境界や層の境界を越えていないか。
- [ ] 同じ tuple や匿名型の形を複数箇所で繰り返す場合、名前付きの型へ切り出しているか。
- [ ] range がコピーを作る場面で、入力サイズや呼び出し頻度に対してコストが妥当か。
- [ ] deconstruction 後の変数名が、元の値の意味を失っていないか。

## テスト観点

- [ ] ガード節は、null、空、範囲外、不正な組み合わせを個別の失敗ケースとしてテストしているか。
- [ ] 条件分岐は、境界値、条件の優先順位、想定外値をテストで押さえているか。
- [ ] `?.` や `??` を使った箇所は、null のときの結果を仕様としてテストしているか。
- [ ] enum に値を追加したとき、コンパイル警告、例外、テスト失敗のいずれかで気付けるか。
- [ ] range / index は、空、1件、ちょうど境界、境界超えのケースを確認しているか。
- [ ] tuple や匿名型で短くした処理は、仕様用語の欠落によってテスト名が曖昧になっていないか。
- [ ] 早期 return の分岐は、副作用が起きないことを確認しているか。
- [ ] flags enum は、単独値、複数組み合わせ、`None`、定義外ビットを確認しているか。

### 使いどころ

この観点は、単なる構文選択ではなく、境界、運用、テスト、レビューで後から判断できる形にしておくために扱います。新規開発では最初から方針を決め、既存コードでは事故が起きやすい境界から段階的に直します。

### 避けたい例

```csharp
// 例: 境界の責務や失敗時の扱いを呼び出し側の暗黙知にしている。
var value = input.ToString();
Save(value);
```

この書き方は小さなサンプルでは動きますが、値の追加、設定変更、外部入力、再試行、キャンセル、ログ出力、テスト実行時に意図しない挙動を隠しやすくなります。

### 改善例

```csharp
public static Result<ValidatedValue> Create(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
    {
        return Result.Invalid("入力が空です。");
    }

    return Result.Success(new ValidatedValue(input.Trim()));
}
```

境界で正規化し、失敗理由を型または戻り値で表すと、レビューとテストで確認する場所が明確になります。例外にするか `Result` にするかは、呼び出し側が復旧できるか、業務エラーとして利用者に返すかで判断します。

### レビュー観点

- 追加・変更時に未対応ケースを隠さず検知できるか。
- 外部入力、DB、JSON、設定、環境差分などの境界で正規化と検証を行っているか。
- 失敗、キャンセル、timeout、再試行、部分成功の扱いがテスト名から読み取れるか。
- ログや例外に機密値、巨大データ、内部実装を出しすぎていないか。
- パフォーマンス改善は計測に基づき、読みやすさを壊すだけの過剰最適化になっていないか。
