# Part 2: 基本文法の実務イディオム

基本文法は、書けるだけならすぐに通過できる領域です。実務では、同じ `if`、`switch`、代入、null 処理でも、「失敗条件が先に読めるか」「値の意味が型や名前に表れているか」「変更に強い境界になっているか」で保守性が大きく変わります。

この Part では、C# の基本構文を、レビューで判断しやすい実務イディオムとして整理します。C# 8 以降の pattern matching、nullable、range、index、tuple なども、便利さだけでなく読みやすさと乱用しない基準を含めて扱います。

各ページでは、概要、判断基準、使いどころ、良い例、避けたい書き方、改善例、レビュー観点、テスト観点をそろえています。構文の紹介ではなく、実務コードで「ここはこの形でよい」と判断するための材料として読みます。

## このPartで身につくこと

- 初期化、代入、ガード節を使い、正常系と異常系の読み順を整えられる。
- `if`、`switch`、pattern matching を、条件の複雑さとドメイン表現に応じて使い分けられる。
- null 合体演算子、null 条件演算子、nullable annotation を過不足なく使える。
- `enum` と flags enum の限界を理解し、必要ならクラスやValue Objectへ移せる。
- range、index、tuple、deconstruction、匿名型を、局所的な読みやすさのために使える。
- 短く書くことと、意図が残ることのバランスをレビューできる。
- 境界値、未対応値、null、空入力など、基本構文に潜む失敗ケースをテスト観点として洗い出せる。

## 重要トピック

- ガード節はネストを減らすだけでなく、処理できない理由を早く示すために使う。
- 例外、早期 return、既定値補完は、呼び出し側の契約に合わせて使い分ける。
- pattern matching は強力ですが、条件が増えすぎるとビジネスルールの置き場所が曖昧になる。
- null 処理は、`?.` や `??` で短くする前に、そもそも null を許すべき値かを判断する。
- `enum` は分類には向きますが、振る舞いや追加データを持ち始めたら別設計を検討する。
- tuple や匿名型は局所的な一時値に向き、公開APIや長く生きるデータ構造には慎重に使う。

## 実務での使いどころ

日々の業務コードで最も頻繁に触る判断基準です。入力検証、コントローラーやユースケースの入口、データ変換、状態判定、検索条件の組み立て、簡単な集計処理などで使います。

特にレビューでは、「もっと短く書けるか」よりも、「次に読む人が条件を間違えずに追えるか」を優先します。新しい構文を使う場合も、チームの標準、対象ランタイム、IDEの補助、テストの書きやすさと合わせて判断します。

## よくある落とし穴

- ガード節を増やしすぎて、メソッドの入口が例外と早期 return だらけになる。
- 複雑な条件式を1行に詰め込み、何を判定しているのか分からなくする。
- `?.` と `??` を重ねて、null が発生した事実を隠してしまう。
- `enum` に表示名、外部コード、振る舞いを詰め込みすぎ、分岐だらけのコードを増やす。
- flags enum で排他的な状態や不正な組み合わせを表してしまう。
- tuple の要素名や順番に依存し、後から読んだときに値の意味が追えなくなる。
- range / index の短さに寄せすぎ、空や少数件の入力で壊れる。

## レビュー観点

- 条件分岐は、正常系、異常系、境界条件のどれを扱っているか読み取れるか。
- null を許す値と許さない値が、型、名前、ガード節で一貫しているか。
- `switch` や pattern matching は、網羅性と将来の追加に強い形になっているか。
- `enum` の値追加時に、修正漏れしやすい分岐が散らばっていないか。
- tuple、匿名型、deconstruction は、局所的な読みやすさに貢献しているか。
- テスト名から、境界値、未対応値、null、空入力の期待動作が読み取れるか。
