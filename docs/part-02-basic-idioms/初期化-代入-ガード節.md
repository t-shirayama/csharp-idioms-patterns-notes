# 初期化、代入、ガード節

## 概要

初期化子、コレクション初期化子、collection expression、ガード節、早期 return は、処理の意図を上から順に読ませるための道具です。特に引数検証は、後続処理の前提を明示する境界として扱います。

実務では「短く初期化できるか」よりも、「いつ有効な状態になるか」「失敗したときに呼び出し側が直せるか」を重視します。入口で前提を確定し、メイン処理では業務上の流れだけを追える形にすると、レビューとテストの焦点が合わせやすくなります。

## 判断基準

初期化は「不完全な状態をどれだけ短く閉じ込められるか」で考えます。オブジェクト初期化子は単純なデータ組み立てに向きますが、複数プロパティの組み合わせに制約があるならコンストラクタやファクトリで守る方が安全です。

ガード節は正常系を読みやすくしますが、何でもメソッド先頭に並べればよいわけではありません。検証が多い場合は、入力モデル、validation 層、Value Object の導入で責務を分けることを検討します。

代入は、値の由来が読み取れる範囲でまとめます。途中で何度も再代入される変数は、条件分岐ごとの状態を追う負担が増えます。後から値を書き換える必要がないなら `var` と初期化を近づけ、再代入が必要なら理由が分かる名前やスコープに閉じます。

ガード節で例外にするか、早期 return にするかは、呼び出し側の期待で決めます。契約違反なら例外、対象データが処理不要なだけなら早期 return が自然です。失敗とスキップを同じ形にすると、ログやテストで原因を見分けにくくなります。

## 使いどころ

- DTO や設定値の組み立てでは、オブジェクト初期化子で「どの値を埋めるか」をまとめる。
- 要素数が少ない固定的なリストでは、collection expression を使うと意図が短くなる。
- 失敗条件はメイン処理の前にガード節として置き、正常系のインデントを浅くする。
- 処理対象外のデータは、例外ではなく早期 return で静かに抜ける選択肢も持つ。
- 入力コレクションを保持する型では、防御的コピーが必要かを入口で決める。

```csharp
public Invoice CreateInvoice(Customer customer, IReadOnlyList<InvoiceLine> lines)
{
    ArgumentNullException.ThrowIfNull(customer);
    ArgumentNullException.ThrowIfNull(lines);

    if (lines.Count == 0)
    {
        throw new ArgumentException("請求行は1件以上必要です。", nameof(lines));
    }

    return new Invoice
    {
        CustomerId = customer.Id,
        Lines = [.. lines],
        Status = InvoiceStatus.Draft
    };
}
```

処理不要を表す早期 return は、失敗とは分けて読めるようにします。

```csharp
public void SendReminder(Invoice invoice)
{
    ArgumentNullException.ThrowIfNull(invoice);

    if (invoice.Status is not InvoiceStatus.Overdue)
    {
        return;
    }

    reminderSender.Send(invoice.CustomerId, invoice.Id);
}
```

## 良い例

ガード節で前提を確定し、以降の処理では null や空を気にせず読めるようにします。初期化時にコピーを作ると、呼び出し元のリスト変更が内部状態へ影響することを避けられます。

```csharp
public sealed class Invoice
{
    public required CustomerId CustomerId { get; init; }
    public required IReadOnlyList<InvoiceLine> Lines { get; init; }
    public InvoiceStatus Status { get; init; } = InvoiceStatus.Draft;
}

var invoice = new Invoice
{
    CustomerId = customer.Id,
    Lines = [.. lines]
};
```

複数プロパティの整合性が必要な場合は、オブジェクト初期化子ではなくファクトリに寄せます。呼び出し側に「この組み合わせでよいか」を覚えさせないことが目的です。

```csharp
public static Invoice CreateDraft(CustomerId customerId, IReadOnlyList<InvoiceLine> lines)
{
    ArgumentNullException.ThrowIfNull(lines);

    if (lines.Count == 0)
    {
        throw new ArgumentException("請求行は1件以上必要です。", nameof(lines));
    }

    return new Invoice(customerId, [.. lines], InvoiceStatus.Draft);
}
```

## 避けたい書き方

- null や空リストの検証を後ろの処理に散らす。
- 代入が何度も分かれ、完成した状態がどこで成立するか分からない。
- ガード節の例外型やメッセージが呼び出し側の判断材料にならない。
- 例外にすべき入力不正と、処理しなくてよい対象外データを同じ `return` で扱う。
- `required` や `init` だけに頼り、業務上の組み合わせ制約を守ったつもりになる。

```csharp
var invoice = new Invoice();
invoice.CustomerId = customer.Id;

if (lines != null)
{
    invoice.Lines = lines.ToList();
}
```

ガード節で雑な例外を投げると、呼び出し側にもログを見る人にも原因が伝わりません。引数不正なら `ArgumentException` 系、状態不正なら `InvalidOperationException` など、失敗の種類に合う例外を選びます。

```csharp
// 避けたい例: どの前提が壊れたのか分からない
if (lines.Count == 0)
{
    throw new Exception("error");
}
```

入力不正を早期 return で隠すと、呼び出し側のバグに気付きにくくなります。

```csharp
// 避けたい例: 必須入力の欠落が成功と同じ経路に見える
if (customer is null || lines is null)
{
    return null;
}
```

## 改善例

状態を段階的に組み立てる必要がある場合は、完成前の変数を外へ出さないようにします。ガード節のメッセージには、呼び出し側が直せる情報を含めます。

```csharp
if (lines.Count == 0)
{
    throw new ArgumentException("請求行は1件以上必要です。", nameof(lines));
}

var invoice = new Invoice
{
    CustomerId = customer.Id,
    Lines = [.. lines],
    Status = InvoiceStatus.Draft
};
```

検証が長くなったら、ガード節を増やす前に入力の概念へ名前を付けます。メソッド本体に残すのは、その処理固有の前提だけにすると読みやすくなります。

```csharp
public Invoice CreateInvoice(CreateInvoiceRequest request)
{
    request.ThrowIfInvalid();

    return Invoice.CreateDraft(request.CustomerId, request.Lines);
}
```

## レビュー観点

- メイン処理に入る前に、null、空、範囲外などの前提が確認されているか。
- 初期化後に不完全なオブジェクトが外へ漏れないか。
- collection expression を使う場合、プロジェクトの C# バージョンで利用できるか。
- ガード節が増えすぎている場合、入力モデルや責務の分離で整理できないか。
- オブジェクト初期化子で不変条件をすり抜けていないか。
- 初期化時に必要なコピーと、過剰なコピーの境界を説明できるか。
- 例外、早期 return、既定値補完の使い分けが、呼び出し側の契約と一致しているか。
- 再代入される変数は、値の変化に業務上の意味があるか。

## テスト観点

- null、空、範囲外、不正な組み合わせを別々のケースとして押さえる。
- 例外型、`paramName`、メッセージが、呼び出し側の調査に足りるかを確認する。
- 早期 return は「処理不要」として仕様化し、副作用が起きないことをテストする。
- 防御的コピーを行う場合、呼び出し元のコレクション変更が内部状態へ影響しないことを確認する。
- ファクトリや Value Object に検証を移した場合、呼び出し側では重複した検証を増やしていないかを見る。
