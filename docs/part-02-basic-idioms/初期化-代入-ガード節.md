# 初期化、代入、ガード節

## 概要

初期化子、コレクション初期化子、collection expression、ガード節、早期 return は、処理の意図を上から順に読ませるための道具です。特に引数検証は、後続処理の前提を明示する境界として扱います。

## 判断基準

初期化は「不完全な状態をどれだけ短く閉じ込められるか」で考えます。オブジェクト初期化子は単純なデータ組み立てに向きますが、複数プロパティの組み合わせに制約があるならコンストラクタやファクトリで守る方が安全です。

ガード節は正常系を読みやすくしますが、何でもメソッド先頭に並べればよいわけではありません。検証が多い場合は、入力モデル、validation 層、Value Object の導入で責務を分けることを検討します。

## 使いどころ

- DTO や設定値の組み立てでは、オブジェクト初期化子で「どの値を埋めるか」をまとめる。
- 要素数が少ない固定的なリストでは、collection expression を使うと意図が短くなる。
- 失敗条件はメイン処理の前にガード節として置き、正常系のインデントを浅くする。

```csharp
public Invoice CreateInvoice(Customer customer, IReadOnlyList<InvoiceLine> lines)
{
    ArgumentNullException.ThrowIfNull(customer);
    ArgumentNullException.ThrowIfNull(lines);

    if (lines.Count == 0)
    {
        throw new ArgumentException("請求行は1件以上必要です。", nameof(lines));
    }

    return new Invoice
    {
        CustomerId = customer.Id,
        Lines = [.. lines],
        Status = InvoiceStatus.Draft
    };
}
```

## 良い例

ガード節で前提を確定し、以降の処理では null や空を気にせず読めるようにします。初期化時にコピーを作ると、呼び出し元のリスト変更が内部状態へ影響することを避けられます。

```csharp
public sealed class Invoice
{
    public required CustomerId CustomerId { get; init; }
    public required IReadOnlyList<InvoiceLine> Lines { get; init; }
    public InvoiceStatus Status { get; init; } = InvoiceStatus.Draft;
}

var invoice = new Invoice
{
    CustomerId = customer.Id,
    Lines = [.. lines]
};
```

## 避けたい書き方

- null や空リストの検証を後ろの処理に散らす。
- 代入が何度も分かれ、完成した状態がどこで成立するか分からない。
- ガード節の例外型やメッセージが呼び出し側の判断材料にならない。

```csharp
var invoice = new Invoice();
invoice.CustomerId = customer.Id;

if (lines != null)
{
    invoice.Lines = lines.ToList();
}
```

ガード節で雑な例外を投げると、呼び出し側にもログを見る人にも原因が伝わりません。引数不正なら `ArgumentException` 系、状態不正なら `InvalidOperationException` など、失敗の種類に合う例外を選びます。

```csharp
// 避けたい例: どの前提が壊れたのか分からない
if (lines.Count == 0)
{
    throw new Exception("error");
}
```

## 改善例

状態を段階的に組み立てる必要がある場合は、完成前の変数を外へ出さないようにします。ガード節のメッセージには、呼び出し側が直せる情報を含めます。

```csharp
if (lines.Count == 0)
{
    throw new ArgumentException("請求行は1件以上必要です。", nameof(lines));
}

var invoice = new Invoice
{
    CustomerId = customer.Id,
    Lines = [.. lines],
    Status = InvoiceStatus.Draft
};
```

## レビュー観点

- メイン処理に入る前に、null、空、範囲外などの前提が確認されているか。
- 初期化後に不完全なオブジェクトが外へ漏れないか。
- collection expression を使う場合、プロジェクトの C# バージョンで利用できるか。
- ガード節が増えすぎている場合、入力モデルや責務の分離で整理できないか。
- オブジェクト初期化子で不変条件をすり抜けていないか。
- 初期化時に必要なコピーと、過剰なコピーの境界を説明できるか。
